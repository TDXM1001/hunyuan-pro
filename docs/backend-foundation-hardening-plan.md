# 后端底座打磨方案

> 方法：按 `karpathy-methodology` 执行，先理解现状，再收敛目标；少做泛化口号，多做可验证的工程闭环。

## 1. 结论

当前后端可以支撑一次底座化改造，不需要推倒重来。

它已经具备系统管理、文件、消息、邮件、缓存、任务、日志、字典、配置、验证码、重复提交保护等基础能力；真正需要补的是平台边界、统一约束、关键公共能力的生产级闭环。

本轮目标不是增加很多业务功能，而是把后续业务接入所依赖的公共能力磨硬，形成一套可复用、可验证、可演进的后端底座。

## 2. 现状判断

### 已具备的基础

- 工程结构已有 `hunyuan-base` 与 `hunyuan-admin` 两层：`base` 承载公共支撑能力，`admin` 承载后台系统与业务入口。
- `hunyuan-base` 已包含 `file`、`message`、`mail`、`dict`、`config`、`operatelog`、`loginlog`、`redis`、`captcha`、`job`、`repeatsubmit`、`securityprotect` 等支撑模块。
- `hunyuan-admin` 的 `system` 下已有 `employee`、`role`、`menu`、`department`、`position`、`login`、`message`、`support` 等系统管理模块。
- 文件模块已有本地存储与 S3 兼容对象存储切换，支持上传、下载、URL 获取、文件记录查询。
- 消息模块已有站内信发送、查询、未读数、已读更新和后台管理接口。
- 全局异常、统一返回体、操作日志、重复提交保护等平台机制已经存在。

### 主要短板

- `sms` 当前只有包目录，没有实际 Java 文件，短信管理还没有形成可调用的底座能力。
- 文件存储缺少完整一致性闭环，例如上传成功但数据库落库失败后的补偿与清理。
- 公共能力之间的边界还不够硬，后续业务接入时容易直接穿透底层实现。
- 权限码、审计字段、错误码、幂等策略、重试策略需要形成统一规范。
- 验证体系偏弱，需要为关键底座链路补“可证明能用”的测试与验收脚本。

## 3. 底座边界

本轮把后端能力划为三层，后续业务只能依赖清晰入口，不能随意反向穿透。

### Core

核心系统管理能力，支撑所有业务模块。

- 用户与登录
- 员工与组织
- 角色与权限
- 菜单与按钮权限
- 部门与岗位
- 字典与系统配置
- 登录日志与操作日志
- 数据权限

### Support

可被业务复用的公共业务支撑能力。

- 文件存储
- 短信
- 站内信
- 邮件
- 验证码
- 序列号
- 定时任务
- 帮助文档
- 变更日志
- 数据变更追踪

### Infra

基础设施适配与运行保障能力。

- MySQL
- Redis / Redisson
- 本地文件存储
- S3 兼容对象存储
- 缓存抽象
- 全局异常
- 统一响应
- 幂等与重复提交保护
- 健康检查
- 可观测性与审计

## 4. 本轮目标

### 必须完成

- 定义底座模块边界和接入规则。
- 补齐短信模块最小可用闭环。
- 强化文件存储一致性与对外接口。
- 收口统一异常、返回、权限码、审计、幂等规范。
- 为文件、短信、消息、登录、权限建立最小验证集。

### 应该完成

- 建立公共能力接口清单，明确哪些 Service 可以被业务调用。
- 为对象存储、本地存储、短信供应商配置环境隔离规则。
- 梳理系统管理菜单与权限码，避免后续重复命名。
- 输出一份后续业务模块接入模板。

### 可以后置

- 完整监控面板。
- 多短信供应商自动降级。
- 大文件分片上传。
- 消息队列化投递。
- 全量链路追踪。

## 5. 模块打磨方案

### 5.1 文件存储

目标：把文件能力从“能上传”打磨成“业务可信赖的文件底座”。

工作项：

- 统一上传、下载、预览、删除、URL 获取接口。
- 明确 `public` 与 `private` 文件访问规则。
- 增加上传失败、落库失败、删除失败的补偿策略。
- 增加文件大小、类型、扩展名、MIME、危险内容校验规则。
- 增加业务引用关系设计，避免文件被误删。
- 输出本地存储和对象存储的配置说明。

验收标准：

- 本地存储模式可上传、下载、预览、删除。
- 对象存储模式可上传、生成临时访问 URL、下载。
- 上传成功但落库失败时不会留下不可追踪文件。
- 私有文件不会直接暴露永久公网 URL。
- 文件查询可按名称、类型、创建人、时间过滤。

### 5.2 短信管理

目标：把 `sms` 从预留目录补成可接入验证码、通知、审批提醒的公共能力。

工作项：

- 建立短信模板表、发送记录表、供应商配置表。
- 建立 `SmsService` 统一入口，业务只调用统一接口。
- 建立供应商适配层，例如 `SmsProvider` 接口。
- 支持验证码短信、通知短信两类最小场景。
- 支持发送前参数校验、频率限制、幂等键、失败记录。
- 支持发送状态、失败原因、重试次数、供应商回执。

验收标准：

- 管理端可以维护短信模板。
- 业务端可以调用统一接口发送短信。
- 每次发送都有记录，可查询状态和失败原因。
- 同一手机号和场景有频率限制。
- 测试环境可以使用 mock provider，不依赖真实短信费用。

### 5.3 消息与邮件

目标：把站内信和邮件沉淀为统一通知能力的一部分。

工作项：

- 统一通知模板命名规范。
- 补消息发送幂等键，避免重复投递。
- 增加消息删除权限边界，避免越权删除。
- 邮件模板补版本与禁用策略说明。
- 明确通知渠道：站内信、邮件、短信。

验收标准：

- 后台可以发送站内信。
- 用户只能查询和标记自己的消息。
- 同一业务事件不会重复产生多条相同通知。
- 邮件模板缺失、禁用、渲染失败时有明确错误。

### 5.4 系统管理

目标：把系统模块作为后续业务权限、组织、菜单的稳定入口。

工作项：

- 梳理用户、员工、角色、菜单、部门、岗位的数据关系。
- 统一权限码命名规范：`module:resource:action`。
- 明确菜单权限与接口权限的映射规则。
- 梳理数据权限策略，确定默认使用方式。
- 输出业务模块接入系统权限的模板。

验收标准：

- 新业务模块可以按模板创建菜单、按钮、接口权限。
- 角色分配后权限即时生效或有明确缓存刷新机制。
- 数据权限规则有示例并可复用。

### 5.5 平台治理

目标：把“写法一致”变成工程约束，而不是靠记忆。

工作项：

- 统一返回体使用规范。
- 统一异常抛出与错误码注册规则。
- 统一参数校验风格。
- 统一操作日志注解使用规则。
- 统一重复提交与幂等策略。
- 建立关键链路测试清单。

验收标准：

- 新 Controller 有固定响应、异常、权限、日志模式。
- 业务错误与系统错误可区分。
- 关键写操作有审计日志或明确豁免理由。
- 重复提交保护有示例和测试。

## 6. 里程碑

### M1：现状固化与边界定义

产出：

- 底座能力清单。
- Core / Support / Infra 边界说明。
- 现有模块风险清单。
- 第一批验收用例。

完成标准：

- 能清楚回答“哪些模块属于底座，哪些模块属于业务”。
- 能清楚回答“业务模块应该调用哪些公共入口”。

### M2：文件存储硬化

产出：

- 文件统一接口。
- 文件一致性补偿策略。
- 文件权限与访问说明。
- 文件链路测试。

完成标准：

- 文件模块可以作为后续业务附件、头像、导入导出的统一底座。

### M3：短信模块成型

产出：

- 短信模板、记录、供应商适配层。
- Mock provider。
- 频率限制与幂等策略。
- 管理端接口。

完成标准：

- 登录验证码、业务通知两类场景可以通过统一入口发送。

### M4：平台规范收口

产出：

- 后端开发规范。
- 权限码规范。
- 错误码规范。
- 审计日志规范。
- 新业务模块接入模板。

完成标准：

- 后续新增业务模块不需要重新设计底层通用能力。

## 7. 第一阶段建议

第一阶段只做 M1，不急着写功能。

原因：当前已有能力不少，直接改代码容易越改越散。先把边界和验收标准定住，再动文件和短信，效率会更高。

建议第一阶段任务：

1. 盘点后端所有 `support` 与 `system` 模块。
2. 标记每个模块状态：可用、需硬化、需补齐、可后置。
3. 输出文件模块与短信模块的接口契约。
4. 输出后续业务接入模板。
5. 确定最小验证命令和测试用例。

## 8. 非目标

- 不在本轮重构整个工程结构。
- 不在本轮引入复杂微服务拆分。
- 不在本轮引入消息队列作为强依赖。
- 不在本轮改造前端 UI 体系。
- 不为了“架构漂亮”牺牲当前可运行状态。

## 9. 验证方式

每个模块完成时至少满足三类验证：

- 编译验证：后端 Maven 编译通过。
- 接口验证：关键 Controller 请求返回符合统一响应结构。
- 数据验证：关键写操作落库、审计、异常补偿可查。

推荐命令：

```bash
cd hunyuan-backend
mvn clean test
```

如本地数据库或 Redis 未就绪，可先执行：

```bash
cd hunyuan-backend
mvn clean package -DskipTests
```

## 10. 当前优先级结论

当前最值得投入的顺序是：

1. 底座边界与模块状态盘点。
2. 文件存储一致性硬化。
3. 短信模块最小可用闭环。
4. 系统管理权限与审计规范收口。
5. 关键链路测试与业务接入模板。

一句话判断：当前后端能支撑底座化改造，但还不够硬；本轮应该先收口边界，再补文件和短信两个最容易被后续业务复用的公共能力。
