# BPM 模块 M8：定义演进与受限实例迁移设计

- 日期：2026-07-12
- 状态：当前模块设计
- 优先级：P4
- 总体蓝图：`docs/superpowers/specs/2026-07-11-bpm-enterprise-blueprint.md`
- 前置条件：M1 版本与稳定 ID、M4 运行事实、M7 处置审计稳定

## 1. 结论

M8 让流程升级可预测、可预演、可审计。默认规则始终是新定义只影响新实例；运行实例迁移是受限管理操作，不是发布新版本的自动副作用。

M8 不承诺任意迁移或物理回滚。已经完成的人工审批、已消费业务结果和不可逆外部副作用只能通过新事实和补偿处理。

## 2. 版本与 Diff

Graph Diff 使用稳定 node、edge、scope ID 比较：新增、删除、配置变化、连接变化、策略变化、依赖版本变化和纯布局变化。布局变化不属于运行语义变化。

每个在途实例冻结源定义、业务契约、候选、数据和连接器版本。发布 V2 不能修改 V1 快照或重新解释 V1 历史。

## 3. 迁移分类

首期允许迁移的实例必须同时满足：

- 静止且没有活动人工任务。
- 单路径，没有活动并行/包容审批组。
- 没有待执行 timer、外部等待或子流程。
- 没有已产生且不可逆的外部副作用。
- authored 节点、数据和变量映射完整且可验证。

不满足条件的实例继续旧版本，或经 M7 人工处置和补偿后重新发起。

## 4. 端到端业务线

```text
发布目标版本
-> Graph 语义 Diff
-> 查询受影响实例
-> 按安全规则分类
-> 配置节点/数据映射
-> 预演并输出阻断原因
-> 人工确认
-> 幂等迁移批次
-> 逐实例结果与前后快照
-> 失败补偿或继续源版本
```

## 5. 失败闭环

- 映射不完整、实例状态变化、活动任务、事件或外部副作用均在预演或执行前阻断。
- 迁移使用批次 ID 与实例级幂等键；重复提交不重复迁移。
- 批次允许部分失败，但每个实例必须有明确结果，失败实例保持源版本事实。
- 迁移后保留源快照、目标快照、映射、操作者、原因和引擎命令证据。
- 定义废弃只阻止新发起，不删除历史定义、实例和审计数据。

## 6. 完成定义

1. V2 发布不改变 V1 在途实例，布局 Diff 不触发迁移建议。
2. 合格静止实例可完成 Diff、预演、确认和迁移，并继续正确运行。
3. 有活动任务、timer、外部等待、子流程或副作用的实例被明确阻断。
4. 重复迁移命令不产生第二次迁移，部分失败批次可安全复核和处置。
5. 任一迁移可追溯源/目标版本、映射、前后状态、操作者和补偿结果。
