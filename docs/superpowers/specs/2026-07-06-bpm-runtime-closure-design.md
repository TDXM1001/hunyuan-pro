# BPM 引擎改造取舍与运行闭环设计

## 背景

当前有两套参考资产：

- 前端参考：`E:\my-project\huanyuan-pro-jichu\yudao-ui-admin-vue3-master`
- 后端参考：`E:\my-project\huanyuan-pro-jichu\ruoyi-vue-pro-master`

参考系统已经具备较完整的 BPM 能力，包括流程分类、表单、模型、定义、发起、待办、已办、抄送、流程详情、评论、监听器、表达式、Flowable 扩展、候选人策略、事件发布、消息通知、业务状态回调等。

`hunyuan-pro` 当前已经开始形成自己的 BPM 底座，包括 `hunyuan-bpm` 模块、Flowable 隐藏内核、分类、表单、模型、定义、实例、任务、监听器、SimpleModel 编译、管理端页面雏形。

本设计采用 Karpathy methodology 的 understanding-first 原则：参考系统是能力样本，不是迁移目标。我们要吸收其成熟机制，但最终形成 Hunyuan 自己的流程底座。

## 目标

第一阶段目标是补齐 `hunyuan-pro` 的通用审批运行闭环：

1. 流程模型可以设计、校验、发布。
2. 已发布流程可以被员工发起。
3. 发起后 Flowable 任务可以同步为平台任务投影。
4. 待办人可以看到待办并完成基础审批动作。
5. 审批动作可以推动 Flowable 与平台状态同步变化。
6. 实例、任务、动作日志可以形成基础轨迹。
7. 运行链路可以被单元测试和最小集成测试验证。

## 非目标

第一阶段不做以下能力：

- 加签、减签、会签、或签、复杂多实例审批。
- 打印模板、流程报表、流程运营大屏。
- 通用表达式平台、复杂条件设计器。
- 通用 HTTP 触发器和外部回调执行器。
- 脚本监听器、class listener、任意表达式 listener。
- 独立 BPM 门户或第二套前端页面体系。
- 将参考仓的权限、用户、部门、岗位、角色模型整体搬入 Hunyuan。

## 设计结论

推荐方案是：

`Hunyuan 原生 BPM 底座 + 借鉴 Ruoyi/Yudao 关键机制`

最终架构定位为：

`Flowable hidden kernel + Hunyuan BPM domain model + Hunyuan employee/org integration + Hunyuan UI shell`

含义如下：

- Flowable 只作为 `hunyuan-bpm` 内部隐藏内核。
- 参考系统只作为能力样本和实现线索。
- 对外 API、数据库投影、前端页面、员工组织身份全部使用 Hunyuan 自己的语义。
- 业务模块通过 Hunyuan BPM API 发起流程、查询状态、接收事件，不直接依赖 Flowable 原生对象。

## 方案比较

### 方案 A：整套迁移 Yudao/Ruoyi BPM

优点：

- 能力覆盖最快。
- 参考系统前后端契约相对完整。

缺点：

- 会把参考系统的用户、权限、部门、岗位、表单、页面风格、Flowable 扩展方式一起带入。
- 与 `hunyuan-pro` 当前系统管理、员工体系、页面标准容易冲突。
- 后续维护会变成“修一套外来系统”。

结论：不推荐。

### 方案 B：只做极简审批流

优点：

- 实现简单，边界干净。
- 首期验证成本低。

缺点：

- 无法承接后续真实流程中心需求。
- 会浪费参考系统中已经成熟的候选人、事件、任务、轨迹等设计经验。

结论：可作为短期 demo，但不适合作为 BPM 底座方向。

### 方案 C：分层吸收，先闭环再增强

优点：

- 保留 Hunyuan 自身架构控制权。
- 能逐步吸收参考系统中真正成熟的能力。
- 每一步都可以独立验证，风险可控。

缺点：

- 需要先做架构取舍，不能机械复制。
- 第一阶段要重点补齐运行投影和状态同步。

结论：推荐。

## 第一阶段范围

第一阶段只做通用审批运行闭环，覆盖以下模块。

### 定义发布

保留当前 `SimpleModel -> validate -> compile BPMN -> deploy Flowable -> definition snapshot` 主线。

需要继续确保：

- 发布后的 definition 不可变。
- 老版本进入历史状态。
- 发起只能使用当前可发起版本。
- 表单、分类、节点规则在发布时形成快照。

### 实例发起

员工通过已发布定义发起流程。

发起时需要：

- 校验 definition 是当前版本并可发起。
- 生成实例编号。
- 调用 Flowable 启动流程实例。
- 写入 `t_bpm_instance` 平台投影。
- 保存 `initial_form_data_snapshot_json` 与 `current_form_data_snapshot_json`。
- 记录发起人、发起部门、业务标识。

### 任务投影

这是第一阶段最关键的补齐项。

Flowable 产生用户任务后，必须同步生成 `t_bpm_task` 平台投影。

任务投影至少包含：

- 平台实例 ID。
- Flowable task ID。
- 定义 ID、节点 ID、节点 key。
- 实例编号、实例标题。
- 发起人快照。
- 当前处理人快照。
- 任务状态、任务结果。
- 到达时间、完成时间。

推荐实现方式：

- 借鉴参考后端的 Flowable task listener 和 task event listener 思路。
- 在 `hunyuan-bpm` 内部新增任务同步协调服务。
- Flowable 原生对象不暴露到控制器和外部模块。

### 审批动作

第一阶段支持四类动作：

- 通过。
- 拒绝。
- 退回发起人。
- 转办。

每个动作都需要同时处理：

- Flowable 任务动作。
- `t_bpm_task` 状态更新。
- `t_bpm_instance` 状态更新。
- `t_bpm_task_action_log` 动作日志。

第一阶段暂不做加签、减签、会签、或签。

### 实例状态

实例状态以 Hunyuan 平台表为查询真相，Flowable 作为运行内核。

基础状态：

- 流转中。
- 待重新提交。
- 已结束。
- 已取消。

基础结果：

- 审批通过。
- 审批拒绝。
- 发起人取消。
- 管理员取消。

### 轨迹与详情

第一阶段先形成基础轨迹：

- 发起记录。
- 任务到达。
- 通过、拒绝、退回、转办。
- 操作人、操作时间、意见。

前端可以先以实例详情抽屉或详情页承载，不引入打印、报表、复杂流程图分析。

## 候选人策略

第一阶段只支持三类候选人：

- `EMPLOYEE`：指定员工。
- `DEPARTMENT_MANAGER`：部门主管。
- `ROLE`：角色成员。

候选人解析由 `hunyuan-admin` 对接当前员工、部门、角色体系实现。

暂不引入：

- 岗位。
- 用户组。
- 多级主管。
- 发起人自选。
- 表单字段动态选人。
- 表达式选人。

这些能力可以作为第二阶段候选人策略增强。

## 前端范围

第一阶段前端分为管理端和员工端。

管理端保留当前 BPM 管理面：

- 分类。
- 表单。
- 模型。
- 定义。
- 实例。
- 任务。
- 监听器目录。

员工端需要补齐运行入口：

- 可发起流程。
- 发起流程。
- 我的申请。
- 我的待办。
- 我的已办。
- 流程详情。

前端借鉴 Yudao 的流程详情、时间线、操作按钮、评论列表结构，但页面外壳遵循 Hunyuan 当前页面标准。

## 后端范围

后端第一阶段重点补齐：

- Flowable 任务同步到平台任务表。
- 任务完成后平台状态同步。
- 实例结束后结果状态同步。
- 退回发起人的状态闭环。
- 转办处理人快照更新。
- 动作日志补全。
- 最小消息通知预留。

暂不做：

- 通用 HTTP 触发器。
- 复杂消息模板。
- 表达式执行器。
- 跨业务 outbox。

## 与业务模块的关系

第一阶段只提供通用流程能力，不绑定具体业务模块。

业务模块接入流程时，只能通过 BPM API：

- 发起流程。
- 查询流程状态。
- 接收流程结果事件。
- 根据 `businessType / businessId / businessKey` 关联业务单据。

业务模块不得直接依赖 Flowable 原生 API。

## 参考系统吸收清单

第一阶段优先吸收：

- 任务服务中的基础审批动作语义。
- Flowable task listener / process listener 的同步思路。
- 流程实例事件发布思路。
- 业务单据与流程实例绑定方式。
- 流程详情页的信息结构。

第一阶段不吸收：

- 完整 Yudao 页面布局。
- 完整候选人策略全集。
- 完整监听器、表达式、HTTP 触发器体系。
- 打印模板和报表。
- OA 请假业务实现。

## 验证标准

第一阶段完成时必须能验证：

1. 发布一个两节点审批流程。
2. 员工 A 发起流程实例。
3. 员工 B 能在待办中看到第一节点任务。
4. 员工 B 审批通过后，第二节点任务生成。
5. 员工 C 审批通过后，实例结束并记录审批通过结果。
6. 拒绝动作能结束实例并记录拒绝结果。
7. 退回发起人能让实例进入待重新提交状态。
8. 转办后新处理人能看到待办。
9. 实例列表、任务列表、动作日志能查到一致状态。
10. Flowable 原生对象不泄漏到外部 API。

## 推荐实施顺序

1. 锁定后端运行状态枚举、动作类型、任务投影字段。
2. 增加 Flowable 任务同步协调服务。
3. 补齐发起后首个任务投影。
4. 补齐通过、拒绝、退回、转办的状态同步。
5. 补齐实例结束事件和结果状态。
6. 补齐动作日志查询和基础详情接口。
7. 补齐员工端待办、已办、我的申请接口。
8. 补齐前端员工端运行页面。
9. 增加后端单元测试和最小集成测试。
10. 增加前端 contract 测试和 typecheck。

## 风险

### 任务投影不同步

如果 Flowable 任务和平台任务表不同步，前端待办会失真。

应对：

- 第一阶段把任务投影列为最高优先级。
- 用测试覆盖发起、完成、转办后的平台表状态。

### 候选人策略过早膨胀

如果第一阶段引入全部候选人策略，规则复杂度会失控。

应对：

- 只保留三类候选人。
- 复杂策略进入第二阶段。

### 参考系统语义侵入

如果直接复用 Yudao/Ruoyi 命名和页面结构，会污染 Hunyuan 自己的系统边界。

应对：

- API、VO、表名、页面外壳使用 Hunyuan 语义。
- 参考代码只作为实现线索。

### Flowable 原生对象外泄

如果业务模块直接依赖 Flowable，后续替换、升级、修复都会困难。

应对：

- Flowable 只存在于 `hunyuan-bpm` 内部 adapter/gateway。
- 外部模块只看 Hunyuan BPM API。

## 开放问题

第一阶段仍需在实施计划中落定：

- 任务投影使用 Flowable listener 还是服务层同步为主。
- 实例结束状态由 Flowable 事件驱动还是审批动作服务同步。
- 退回发起人是否需要生成“重新提交”任务，还是由实例状态和发起人页面承载。
- 员工端流程页面放在 `hunyuan-system` 还是新增更明确的运行端入口。

## 自检

- 本设计没有要求整套迁移参考仓。
- 本设计只聚焦第一阶段运行闭环。
- 本设计明确排除了加签、打印、报表、复杂表达式、HTTP 触发器。
- 本设计明确了任务投影是首要补齐点。
- 本设计保留 Hunyuan 自身模块、组织、页面和 API 边界。
