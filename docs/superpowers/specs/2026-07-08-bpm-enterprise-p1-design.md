# Hunyuan BPM 企业级流程引擎 P1 设计

## 结论

P1 的目标不是继续零散补页面或审批按钮，而是把当前已经闭环的 `hunyuan-bpm` 升级为 **Hunyuan 原生企业级流程平台最小可用版**。

推荐路线采用 **平台治理优先 + 一个真实业务样板验收**：

- 先把 BPM 自身的平台边界、定义治理、运行治理、审计、通知、业务接入契约做稳。
- 再用一个最小的费用/采购申请样板验证业务接入，而不是让 BPM 底座依赖某个业务模块。
- 参考 `yudao-ui-admin-vue3-master` 和 `ruoyi-vue-pro-master` 的能力面与机制，但不迁移它们的路由、接口、命名、页面壳、权限模型或 Flowable 暴露方式。

## 当前基线

当前 `hunyuan-pro` 已经不是空白 BPM。P1 必须沿着现有实现演进，而不是重新移植参考项目。

已完成能力：

- 后端已有独立 `hunyuan-bpm` 模块。
- Flowable 作为 BPM 模块内部隐藏内核接入。
- 已有 `t_bpm_category`、`t_bpm_form`、`t_bpm_model`、`t_bpm_definition`、`t_bpm_definition_node`、`t_bpm_instance`、`t_bpm_task`、`t_bpm_task_action_log`、`t_bpm_instance_copy` 九张核心表。
- 已有 SimpleModel 草稿、校验、模拟、编译 BPMN、发布 definition snapshot 的主链路。
- 员工侧运行入口已覆盖可发起、发起草稿、我的申请、重提草稿、我的待办、我的已办、我的抄送、实例详情。
- 任务动作已覆盖通过、拒绝、退回发起人、转办、重提、取消、手工抄送。
- 实例详情已统一展示基础信息、当前待办、表单快照、动作轨迹。
- 已有测试保护 Flowable 不外溢到公开控制器、VO、前端 API 类型。

当前缺口：

- 定义治理还偏 P0：流程版本、停用、可发起范围、发布前校验、发布后差异说明还需要产品化。
- 运行治理还偏基础闭环：管理员干预、任务追回、委派、加签、减签、超时策略尚未进入 Hunyuan 语义。
- 集成可靠性不足：通知、事件、业务回调还缺少持久化失败记录、重试、幂等、补偿与可观测入口。
- 业务接入契约还未成型：业务模块如何发起、如何读取状态、如何接收结果、如何处理失败需要统一。
- 前端流程中心已经可用，但企业级治理视角下还需要定义中心、运行中心、监控中心的清晰分工。

## 参考项目取舍

### 借鉴内容

从参考前端借鉴：

- 流程中心的信息架构：流程定义、模型设计、发起流程、我的待办、我的已办、我的抄送、实例详情、流程报表。
- 实例详情页的组织方式：表单数据、流程图/节点视图、审批记录、评论/附件、操作区分层展示。
- 简易流程设计器的产品思路：业务用户面对 SimpleModel，而不是一开始就面对通用 BPMN 画布。
- 任务页的操作丰富度：通过、拒绝、转办、委派、加签、减签、退回、抄送、打印等能力目录。

从参考后端借鉴：

- 模型发布前进行候选人、表单、节点、BPMN 的一致性校验。
- 任务生命周期集中在 BPM service 层，不把 Flowable 原生对象暴露给业务模块。
- 业务通过 `businessKey` 或等价 Hunyuan 业务引用和流程实例建立关系。
- 事件、监听器、消息、回调形成明确扩展点。
- 候选人解析策略独立成可测试单元。

### 不借鉴内容

P1 不做以下迁移：

- 不迁移 Yudao/RuoYi 的 `/admin-api/bpm/**` 路径。
- 不迁移它们的 VO、DO、枚举名、权限码、菜单结构。
- 不把 Flowable `Task`、`ProcessInstance`、`ProcessDefinition` 暴露给 Hunyuan 公开 API。
- 不把参考项目的通用 HTTP listener 直接开放为 P1 执行能力。
- 不把参考项目前端页面壳照搬到 Hunyuan Design。
- 不为了追齐功能目录而跳过 Hunyuan 员工、部门、角色、菜单、消息、短信、邮件的现有边界。

## P1 总目标

P1 完成后，Hunyuan BPM 应达到以下状态：

1. 平台管理员能管理流程分类、表单、模型、定义版本、发布状态、发起范围。
2. 员工能稳定完成发起、审批、退回重提、转办、抄送、查看轨迹。
3. 管理员能查看实例、任务、动作轨迹，并进行受控干预。
4. 业务模块能通过 Hunyuan BPM 契约发起流程并跟踪流程结果。
5. 通知、事件、业务回调有持久化记录，失败可查、可重试、可补偿。
6. 审批语义以 Hunyuan 名称暴露，Flowable 只作为内部内核。
7. 所有关键能力都有后端单测、前端契约测试或可复现的浏览器验收路径。

## P1 模块分层

### 1. 定义治理中心

职责：

- 流程分类治理。
- BPM 表单治理。
- SimpleModel 设计与校验。
- 流程模型列表与草稿管理。
- 定义发布、历史版本、停用/启用。
- 发布前风险检查和发布后快照固化。

P1 增强重点：

- 发布前校验报告：表单是否可用、节点是否有处理人、变量映射是否在白名单内、通知渠道是否可用。
- 发布差异说明：本次发布相对上一版本变更了哪些节点、表单、候选人规则、通知配置。
- 可发起范围：按员工、部门、角色或全员限定谁能看到并发起流程。
- 定义状态治理：当前版、历史版、可发起、已停用必须语义清楚。

验收标准：

- 同一模型多次发布形成不可变定义版本。
- 历史定义不被草稿修改影响。
- 停用后的定义不能被新发起，但历史实例仍可查看。
- 未通过校验的模型不能发布。

### 2. 运行工作台

职责：

- 可发起流程。
- 发起表单。
- 我的申请。
- 我的待办。
- 我的已办。
- 我的抄送。
- 实例详情。
- 任务详情。

P1 增强重点：

- 实例详情抽屉/页面统一成为运行事实入口。
- 待办动作统一使用同一套动作模型。
- 抄送记录支持已读、详情、来源动作追踪。
- 重提保留初始表单快照与当前表单快照的区别。
- 当前节点摘要、处理人、任务状态、动作轨迹保持一致。

验收标准：

- 发起、通过、拒绝、退回、重提、转办、抄送都能在同一实例详情中解释清楚。
- 员工只能看到自己可见的待办、已办、申请、抄送。
- 管理员可以从实例列表进入同一详情视角，但权限更高。

### 3. 管理干预中心

职责：

- 管理员查看所有实例。
- 管理员查看所有任务。
- 受控执行干预动作。
- 记录每一次人工干预的原因和操作者。

P1 第一批干预动作：

- 管理取消实例。
- 管理转办任务。
- 管理补录抄送。
- 管理重同步任务投影。

P1 暂缓动作：

- 管理跳转节点。
- 管理直接改变量。
- 管理直接改 Flowable 原生运行表。

验收标准：

- 管理动作必须写入 `t_bpm_task_action_log` 或对应审计记录。
- 管理动作必须和员工动作区分 actor 与 action type。
- 管理动作失败不能造成实例表、任务表、Flowable 内核状态互相矛盾。

### 4. 业务接入契约

职责：

- 业务模块通过稳定契约发起流程。
- 业务模块通过稳定契约查询流程状态。
- BPM 通过事件或回调通知业务流程结果。
- BPM 记录业务接入和回调失败。

P1 推荐契约：

- `BpmBusinessStartCommand`
  - `businessType`
  - `businessId`
  - `businessKey`
  - `definitionKey`
  - `startEmployeeId`
  - `formDataJson`
  - `title`
  - `summary`
- `BpmBusinessInstanceStatus`
  - `instanceId`
  - `instanceNo`
  - `businessType`
  - `businessId`
  - `runState`
  - `resultState`
  - `lastActionAt`
- `BpmBusinessResultEvent`
  - `eventId`
  - `instanceId`
  - `businessType`
  - `businessId`
  - `resultState`
  - `payloadJson`
  - `occurredAt`

P1 样板业务：

- 默认使用一个最小费用/采购申请样板验证接入契约。
- 样板只验证业务接入、状态同步、审批结果回写、失败补偿，不把业务字段扩散到 BPM 底座。
- 样板可以放在独立 demo/acceptance 范围，不作为业务模块通用模板强推。

验收标准：

- 样板业务能通过契约发起流程。
- 审批完成后业务侧能拿到通过/拒绝结果。
- 重复回调不会重复改变业务状态。
- 回调失败有记录并可重试。

### 5. 可靠性与可观测

职责：

- 记录 BPM 内部事件。
- 记录通知发送结果。
- 记录业务回调结果。
- 支持失败重试。
- 支持按实例追踪一条流程从发起到结束的关键事件。

P1 建议新增或固化的记录面：

- BPM 事件记录：实例创建、任务创建、任务完成、实例结束、实例取消。
- 通知记录：站内信、短信、邮件的发送请求、结果、失败原因。
- 回调记录：业务回调请求、响应、重试次数、最终状态。
- 命令记录：关键审批命令的请求摘要、执行结果、失败原因。

实现原则：

- 优先复用现有 `message`、`sms`、`mail`、操作日志能力。
- 必要的 BPM 专属事件/回调记录可以进入 `hunyuan-bpm`，但不能把通用消息能力复制一套。
- P1 先支持可查、可重试、可解释，不追求完整分布式事务平台。

验收标准：

- 任意一个实例可以查到关键生命周期事件。
- 任意一次通知失败可以定位到实例、任务、接收人、渠道和失败原因。
- 任意一次业务回调失败可以重试，重复重试不造成业务状态重复推进。

### 6. 审批语义增强

P1 推荐优先级：

1. 委派：当前处理人临时交给他人处理，但责任关系保留。
2. 加签：当前节点增加额外处理人。
3. 减签：从当前节点移除尚未处理的人。
4. 追回：发起人或管理员在特定状态下撤回到可编辑状态。

P1 约束：

- 所有语义使用 Hunyuan 命名。
- Flowable 多实例、委派等原生能力只在内部适配层出现。
- 每个动作必须有 action log。
- 每个动作必须明确是否改变实例状态、任务状态、当前处理人、表单快照。

P1 不做：

- 复杂比例会签。
- 任意节点跳转。
- 任意脚本条件。
- 通用 HTTP 节点执行平台。
- 子流程编排。

## 前端体验原则

P1 前端继续遵守 Hunyuan 现有页面规范：

- 列表页保持安静、密集、可扫描。
- 详情优先使用统一抽屉/侧边界面保留父列表上下文。
- 不照搬 Yudao 页面壳。
- 设计器可借鉴参考项目的信息分组，但控件、路由、接口、状态处理必须 Hunyuan 原生。
- 普通菜单页面不增加重复说明文案。
- 只有一行搜索条件时禁用无意义的展开/收起。

P1 页面目标：

- 定义治理：分类、表单、模型、定义、设计器、发布校验。
- 运行工作台：可发起、我的申请、我的待办、我的已办、我的抄送。
- 管理中心：实例管理、任务管理、事件/回调记录。
- 样板验收：费用/采购申请发起与状态查看。

## 后端边界原则

- `hunyuan-bpm` 是流程平台模块。
- `hunyuan-admin` 提供员工、部门、角色、菜单、权限主体。
- `hunyuan-base` 提供消息、短信、邮件、文件、编号、日志等基础能力。
- 业务模块通过 BPM 契约接入，不直接依赖 Flowable。
- 公开 controller、VO、前端 API 类型不得导入或暴露 `org.flowable`。
- Flowable 网关只允许在 `hunyuan-bpm.engine.internal` 或等价内部包中出现。

## 阶段拆分

### P1.1 定义治理加固

目标：

- 发布前校验报告。
- 发布差异说明。
- 定义停用/启用规则。
- 可发起范围。

验收：

- `mvn -f hunyuan-backend/pom.xml -pl hunyuan-bpm test`
- BPM 前端 API/模块契约测试通过。
- 定义发布、停用、发起范围有后端单测。

### P1.2 运行治理加固

目标：

- 统一实例详情作为运行事实入口。
- 管理员实例/任务管理补齐受控干预。
- 待办动作模型统一。
- 抄送和重提轨迹继续收敛。

验收：

- 发起、通过、拒绝、退回、重提、转办、抄送、管理取消都能在动作轨迹中解释。
- 管理动作单测覆盖权限边界和审计记录。

### P1.3 业务接入与可靠性

目标：

- 建立业务发起、状态查询、结果事件契约。
- 建立业务回调记录与重试机制。
- 建立通知发送记录与失败查询。
- 完成费用/采购申请样板流程。

验收：

- 样板业务能发起流程并收到审批结果。
- 回调失败可查、可重试、幂等。
- 通知失败可查。

### P1.4 审批语义增强

目标：

- 委派。
- 加签。
- 减签。
- 追回。

验收：

- 每个动作都有独立单测。
- 每个动作都进入动作轨迹。
- 每个动作都不暴露 Flowable 原生对象。

## P1 完成定义

P1 可收尾的标准：

- BPM 定义治理、运行工作台、管理干预、业务接入、可靠性记录都有 Hunyuan 原生入口。
- 一个最小费用/采购申请样板完成从业务发起到审批结果回写的闭环。
- 所有公开 API 与前端类型不暴露 Flowable。
- 关键失败路径可查、可重试或有明确补偿说明。
- 后端单测、前端契约测试、类型检查通过。
- 至少一条真实浏览器业务流有验收记录。

## 明确非目标

P1 不做：

- 不做通用 BPMN 画布优先的全能力设计器。
- 不做复杂比例会签、任意跳转、子流程、边界事件、脚本执行平台。
- 不做通用 HTTP 节点执行平台。
- 不做跨业务域的强绑定流程模板。
- 不替代 `hunyuan-base` 的消息、短信、邮件能力。
- 不迁移参考项目的模块结构或接口契约。
- 不把业务样板变成所有业务模块必须复制的模板。

## 理解校验

关键假设：

- 当前 BPM P0 运行闭环已经成立，P1 应在此基础上增强治理和可靠性。
- 企业级不是功能堆叠，而是边界、审计、接入、失败处理、可验证能力同时成立。
- 参考项目的价值在能力面和机制，不在代码迁移。

已验证依据：

- 当前仓库存在 `hunyuan-bpm` 模块、BPM 控制器、Flowable 内部网关、九张核心 BPM 表和相关测试。
- 当前仓库已有运行端闭环验收文档、实例详情验收文档、手工抄送验收文档。
- 参考仓库存在 BPM 模型、定义、任务、实例、候选人策略、事件、监听器、HTTP trigger、前端流程中心等机制。

仍需后续实现时验证：

- 费用/采购申请样板应选择最小接入方式，避免污染生产业务模块。
- 事件/回调记录是否复用既有审计或新增 BPM 专属表，需要在实现计划中结合当前表结构细化。
- 委派、加签、减签、追回应逐项验证 Flowable 内核适配方式，并保持 Hunyuan 公开语义稳定。
