# BPM 模块 M7：运营与治理设计

- 日期：2026-07-11
- 重审日期：2026-07-12
- 状态：当前模块设计
- 优先级：P3
- 总体蓝图：`docs/superpowers/specs/2026-07-11-bpm-enterprise-blueprint.md`
- 前置条件：M4 核心运行稳定；M5/M6 产生完整事件与交付事实

## 1. 结论

M7 让运营人员能够发现、解释和受控处置流程异常，而不是依赖数据库修改、Flowable 控制台或开发人员临时脚本。

M7 负责运营搜索、异常队列、SLA 看板、受控干预、重试、补偿、报表、审计、归档与数据保留。它不重新定义任务、事件或业务契约，只消费 M4-M6 的结构化事实并发出受控命令。

## 2. 运营对象

- 定义版本、实例、任务、审批组和路由决定。
- timer、外部等待、子流程和连接器尝试。
- 业务命令、通知、结果 Outbox、回调、死信和补偿。
- 迁移预演与执行批次。

所有对象使用 Hunyuan ID、状态和关联关系，不从 BPMN XML 或引擎表拼装业务状态。

## 3. 端到端业务线

```text
监测积压 / SLA / 失败事实
-> 告警和统一异常队列
-> 定位业务对象、定义、实例、节点与外部记录
-> 校验管理员权限和数据范围
-> 催办 / 重试 / 补偿 / 暂停 / 终止
-> 复核运行与业务结果
-> 固化处置审计和报表
```

普通业务管理员按组织和流程范围查看；专业流程管理员按定义责任范围处置；平台运维只拥有明确高风险命令。处置权限不自动授予敏感业务字段查看权。

## 4. 处置与安全

- 每项命令定义合法前置状态、幂等键、权限、原因和预期后置状态。
- 高风险迁移、终止、自动终态补偿可配置双人复核。
- 原审批动作、路由事实和外部响应不可覆盖，只能追加处置事实。
- 禁止任意 execution jump、直接改 Flowable 表或伪造任务完成。
- 导出、打印、报表和审计按 M3 字段敏感级别脱敏。
- 数据保留、归档和清理按定义、业务契约、审计法规和法律保留状态执行；不得级联删除仍被实例或证据引用的快照。

## 5. 失败闭环

- 自动重试耗尽后进入明确死信/人工处置状态并指定责任人。
- 处置命令失败保留尝试与原因，可安全重试时继续使用相同幂等边界。
- 无法补偿的外部副作用必须明确标记并升级人工决策，不伪造已恢复状态。
- 监控、投影或报表延迟不得改变审批运行事实，只标记数据新鲜度。

## 6. 完成定义

1. 运营人员可按业务键、定义版本、处理人、状态、SLA、失败码和事件 ID 检索。
2. 每类核心异常至少证明检测、定位、授权处置、恢复/终止和审计闭环。
3. 重试、补偿、终止和高风险动作具备权限、原因、前后快照和幂等性。
4. SLA、积压、处理时长和失败趋势可按定义版本、节点和组织统计。
5. 报表、打印和审计不直接查询 Flowable 表，也不泄露越权字段。
6. 归档与清理不会破坏在途实例、业务证据、迁移来源或法定保留记录。
