## 1. 总体结论

Hunyuan BPM 的目标是建设一个与具体业务解耦、由配置驱动、能够完整治理企业审批过程的流程基座。它不拥有采购、请假、合同、报销等业务主数据，但必须让这些业务通过统一契约关联流程，并保证审批人始终能看到本次审批所依据的业务内容、流程状态和操作证据。

最终边界是：

```text
具体流程由管理员配置
业务单据由业务系统拥有
审批过程与证据由 Hunyuan BPM 管理
Flowable 只承担内部 BPMN 执行
```

新增流程定义只使用正式流程图 `HunyuanProcessDefinitionGraph`。数据库已经清空，因此不建设旧 `SimpleModel` 导入、双格式、双写或兼容发布链。现有 M1、M3、M4 代码只能作为已验证算法、状态语义和测试证据的来源；新数据模型、接口、模块边界和用户体验不受旧类名与旧 JSON 结构约束。

配置决定节点、连线、条件、审批人、审批策略、字段权限、超时、外部等待、子流程、业务绑定、详情布局、事件订阅和结果回传。平台代码只负责保存、校验、编译、执行和治理这些配置，不能为采购、请假等业务写死流程分支。

## 2. 产品服务对象

### 2.1 普通业务管理员

- 使用业务化节点、自动布局和表单式配置创建常见审批流程。
- 配置审批人、条件、字段权限、时限和业务绑定，不接触 BPMN、XML、EL 或 Flowable ID。
- 查看授权范围内的流程运行情况，执行催办或受控处置。

### 2.2 专业流程管理员

- 配置连接关系、条件/并行/包容分支、汇合、多人策略、事件和子流程。
- 使用模拟、Diff、发布预检和版本策略治理流程资产。
- 能查看结构错误与影响分析，但不能注入任意脚本或绕过平台约束。

### 2.3 实施与集成人员

- 登记来源系统、业务契约、员工映射、连接器、事件订阅和环境配置。
- 查看只读编译诊断、authored/compiled 映射、部署结果与迁移影响。
- 不能直接修改部署 XML、Flowable 表或注入 Java Bean。

### 2.4 申请人、审批人和经办人

- 申请人能够发起、补件、撤回并查看流程进度。
- 审批人能够看到审批对象、字段权限、历史动作和当前允许操作。
- 经办人能够完成办理任务，但不能越权执行审批动作。

### 2.5 运营人员与外部系统

- 运营人员通过结构化事实定位异常、重试、补偿和审计，不直接修数据库。
- 外部系统通过固定 API、嵌入组件和可靠事件接入，不直接读写 BPM 表。

这些用户不是三套流程引擎。平台只有一份正式流程图、一套发布版本和一套任务命令；不同角色只拥有不同的编辑深度、诊断信息和操作权限。

## 3. 产品范围与非目标

### 3.1 产品范围

- 申请、审批、经办、抄送、退回、撤回、取消和管理员处置。
- 流程分类、模板、发起/可见范围、任务通知和定义下线。
- 单人、顺序多人、并行会签、或签和比例完成策略。
- 条件、并行、包容分支及结构化汇合。
- 任务 SLA、延迟、登记连接器、外部等待和子流程。
- 内置 Hunyuan 表单与外部业务系统统一接入。
- Hunyuan 工作台、嵌入审批组件和开放任务 API 三种入口。
- 定义版本、运行快照、可靠事件、异常处置和受限实例迁移。

### 3.2 非目标

- 不开放任意 BPMN 元素、任意循环、任意 EL、脚本、Java Bean 或类名调用。
- 不允许流程定义直接保存任意 HTTP URL、明文凭据或数据库查询。
- 不把 BPM 建成采购、合同、请假等业务主数据系统。
- 不允许外部应用凭证代替员工身份处理人工任务。
- 不承诺任意运行中实例都能迁移、回滚或恢复外部不可逆副作用。
- 不通过解析 BPMN XML、Flowable task key 或名称猜测 Hunyuan 业务状态。

## 5. 正式流程定义模型

### 5.1 唯一作者模型

正式保存真相为：

```text
HunyuanProcessDefinitionGraph
  schemaVersion
  rootScopeId
  scopes[]
  nodes[]
  edges[]
  policies
  layout
```

- `schemaVersion` 只表示图契约版本，不等同于草稿修订号、发布版本或编译器版本。
- `scope` 表示主流程和子流程作用域。
- `node` 保存稳定逻辑 ID、所属作用域、节点类型和类型化配置。
- `edge` 保存稳定 ID、来源、目标、端口、分支身份、优先级和受控条件。
- `policies` 保存流程级退回、取消、异常和完成策略。
- `layout` 只保存位置、折叠和视图状态，不参与运行语义。

节点、边和作用域使用系统生成且不可复用的稳定 ID。显示名称可以修改，稳定 ID 不随改名和拖动改变；编译器另行生成 BPMN ID，并保存 authored 元素到 compiled 元素的一对多映射。

### 5.2 节点目录

M1 核心节点包括：开始、结束、审批、办理、抄送、条件分支、并行分支和包容分支。M5 在同一节点注册和版本机制上增加延迟、外部调用/等待和调用子流程。

节点类型定义稳定能力边界，业务值全部来自配置。例如审批节点定义候选来源、完成策略、字段权限和 SLA 能力，但不在代码中写死采购金额、部门名称或领导账号。

### 5.3 图结构约束

正式流程图是受控、可结构化归约的图，不是任意有向图。发布至少校验：

- 每个作用域恰好一个开始节点，并至少存在一个可达结束节点。
- 不存在孤儿、不可达或无法到达结束节点的节点。
- 开始、结束、任务、分叉和汇合的入出边及端口数量符合节点契约。
- 条件、并行、包容分叉与汇合必须成对、同作用域且良好嵌套，不允许交叉汇合。
- 条件和包容分支存在明确默认路径，或配置无匹配时失败关闭策略。
- 普通流程边无环；退回和重提通过受控运行策略表达，不画任意反向边。
- 子流程引用无循环依赖，连接不得跨越作用域边界。
- 并发分支写入同一流程字段时必须有受控合并策略，否则阻断发布。
- 条件只引用冻结路由事实或登记表达式，不保存可执行脚本。

## 6. 设计器与发布治理

设计器围绕同一流程图提供三种能力层：

- 普通模式：业务化节点、节点间新增、自动布局和表单式属性配置。
- 专业模式：显式连接、分支/汇合、事件、子流程和版本策略。
- 实施诊断：只读查看编译 BPMN、元素映射、部署结果和错误定位。

三种模式不能形成三份模型或三套发布链。

发布生命周期为：

```text
草稿 -> 局部校验 -> 全局结构校验 -> 业务引用校验
     -> 样例模拟 -> 语义 Diff -> 发布确认 -> 不可变定义版本
```

发布失败不得产生半发布定义或无法追踪的 Flowable 部署。已发布版本冻结流程图、表单/业务契约、候选策略、连接器引用、编译器版本、BPMN 产物和 authored/compiled 映射。草稿修改不影响已发布定义，布局移动不产生新的语义版本。

## 7. 审批对象与数据边界

每次发起使用 `(sourceSystem, businessType, businessKey)` 唯一定位审批对象，业务键使用字符串，不要求外部系统使用 Long ID。

运行实例必须区分四类数据：

```text
ApprovalSubjectSnapshot
  只读审批依据：标题、摘要、字段、明细、附件、提交人

RoutingFactSnapshot
  经业务 schema 校验后冻结的流程选择与条件路由事实

ProcessWorkingData
  按节点权限可修改的流程数据，如核定金额和补充说明

TaskActionEvidence
  通过、拒绝、退回、意见、签名、附件和操作版本
```

审批对象快照是所有任务入口的最低展示保证。来源系统不可用时，审批人仍能查看发起时依据；实时查询只用于补充当前业务状态，不能覆盖历史快照。

业务对象变化策略由业务契约配置：

- `LOCKED`：审批中禁止修改。
- `VERSIONED`：允许形成新版本，展示差异并通知后续处理人。
- `RESTART_REQUIRED`：关键字段变化后终止当前审批并重新发起。
- `FIELD_CONTROLLED`：仅允许指定字段变化。

BPM 不绕过业务服务修改业务主表，业务系统也不能直接修改流程工作数据或审批证据。

## 8. 身份、任务与统一命令

审批人、经办人、抄送人和管理员统一使用 Hunyuan 员工、组织和权限体系。候选策略可引用员工、角色、部门负责人、发起人、发起人部门负责人和发起时选择字段；扩展来源必须通过登记解析器，不允许脚本表达式。

发布前执行候选模拟和静态预检，运行时解析并冻结候选成员快照。无候选人、成员失效或组织数据冲突必须进入明确异常状态，不能静默跳过节点。

Hunyuan 工作台、嵌入组件和外部页面共享统一动作命令：

```text
TaskActionCommand
  requestId
  taskId / taskVersion
  action
  actorIdentity（服务端从登录会话或外部员工映射注入）
  approvalSubjectVersion
  processWorkingDataVersion
  comment / attachments
  clientChannel
```

客户端不得自报或覆盖处理人身份。服务端依次验证应用身份、从会话/映射生成员工身份、原子查询或占用 `(tenantId, instanceId, requestId)` 命令回执：同 canonical 指纹直接返回既有结果、异指纹冲突；只有新回执才继续校验任务权限、任务状态、数据版本并推进流程。前端只能根据服务端返回的 `allowedActions` 渲染操作，不能自行推导审批规则。

外部用户必须映射到 Hunyuan 员工账号后才能读取或处理人工任务。未映射身份只能触发已授权系统事件。

## 9. 业务接入与可靠事件

M3 拥有业务契约的字段、快照、权限和数据变化语义；M6 负责来源系统登记、契约发布入口、流程绑定、外部认证和可靠交付。二者引用同一 `BusinessContractVersion`，不得各建一套业务契约实体。标准接入由配置中心管理：

```text
SourceSystem
  系统编码、认证、签名、公钥、登记端点、网络策略、启停状态

BusinessContract
  业务类型、键规则、字段 schema、敏感级别、附件规则、详情布局、变更策略

ProcessBinding
  业务类型、组织/场景范围、匹配条件、优先级、流程定义、有效期

EventSubscription
  事件类型、登记端点、签名、重试、死信和权限范围
```

流程匹配条件只能读取 M3 按业务契约校验并冻结的路由事实，使用类型化操作符或登记规则，禁止脚本和实时跨库查询。匹配必须且只能命中一条有效绑定；无匹配、同优先级多匹配或契约版本不兼容时阻断发起，并返回可诊断结果。

跨系统事件采用至少一次投递：Outbox、Inbox、`eventId` 幂等、签名、指数重试、死信和人工补偿。平台不承诺物理意义上的 exactly-once。连接器、等待、回调和业务事件均记录请求/响应摘要、签名结果、实例关联、重试次数和补偿动作。

## 10. 版本、运行治理与实例演进

每个实例至少冻结流程定义版本、业务契约版本、绑定版本、审批对象版本、路由事实版本、候选快照和连接器版本。

运行视图以 Hunyuan 结构化投影展示审批对象、当前状态、authored 运行图、路由决定、审批阶段、数据版本、SLA、外部等待、子流程、动作时间线和补偿记录。

管理员取消、转办、重试、补偿和迁移均要求权限、原因、幂等命令及前后状态审计。管理员获得处置权限不等于可以读取全部敏感业务字段。

定义发布默认只影响新实例。运行实例迁移遵循：

```text
定义 Diff -> 影响分析 -> 映射配置 -> 预演 -> 人工确认
         -> 幂等执行 -> 结果审计 -> 失败补偿
```

首期只允许迁移静止、单路径、无活动人工任务、无并行审批、无计时器、无外部等待、无子流程且无不可逆外部副作用的实例。其他实例继续旧版本或进入人工处置，不允许绕过限制强制迁移。

## 11. M1-M8 模块体系

```text
M1 -> (M2 + M3) -> M4 -> M5 -> M6 -> M7 -> M8
```

| 模块 | 能力面 | 服务对象 | 闭环结果 |
| --- | --- | --- | --- |
| M1 | 流程定义中心 | 普通/专业流程管理员、实施人员 | 从空白流程图到校验、模拟、发布和不可变定义版本 |
| M2 | 身份组织与审批策略 | 流程管理员、审批人 | 从发起/可见范围、候选配置到预检、成员冻结和完成策略 |
| M3 | 审批对象与数据治理 | 申请人、审批人、实施人员 | 从业务契约到审批快照、路由事实、工作数据和动作证据 |
| M4 | 核心审批运行与工作台 | 申请人、审批人、经办人 | 从发起到路由、任务处理、退回/取消和终态审计 |
| M5 | 高级流程运行 | 流程管理员、运营人员 | 时间、外部等待和子流程从配置到可靠推进与恢复 |
| M6 | 配置化业务接入 | 实施人员、外部系统 | 从来源登记到流程绑定、内外任务和可靠结果交付 |
| M7 | 运营与治理 | 业务管理员、专业管理员、审计人员 | 从异常发现到受控处置、补偿、度量和审计 |
| M8 | 定义演进与受限迁移 | 专业管理员、实施人员 | 从版本 Diff 到影响分析、预演和安全迁移 |

M1-M4 共同构成第一个可交付产品基线。M1、M2 或 M3 单独完成时只能说明对应能力面闭环，不能宣称企业审批平台已经可用。M5 先稳定时间、等待和子流程事实，M6 再把稳定能力开放给外部系统；M7-M8 负责长期运营与演进。

M1 与 M2/M3 不形成循环所有权：M1 先冻结 Graph、版本引用和 resolver 端口；M2/M3 分别提供真实策略目录与业务契约目录；三者集成校验通过后，M1 的生产发布门才开放。M1 可以先验证结构和发布机制，但不能在真实引用缺失时发布可发起的业务定义。

## 12. 模块业务线与完成定义

### 12.1 M1 流程定义中心

业务线：创建流程资产 -> 配置正式流程图 -> 校验结构与引用 -> 样例模拟 -> 语义 Diff -> 发布 -> 冻结 Graph/BPMN/映射。

失败闭环：非法图定位到节点、边或属性；并发保存返回草稿版本冲突；发布失败不留下半发布定义；部署产物无法反查 authored 元素时禁止发布。

完成定义：三类管理员均使用同一 Graph API；核心节点可配置和发布；新数据库不存在旧作者模型字段、解析分支或双写逻辑；已发布版本不可修改；布局变化不产生语义版本。

### 12.2 M2 身份组织与审批策略

业务线：配置候选来源和完成策略 -> 发布预检/模拟 -> 运行解析 -> 成员冻结 -> 单人/顺序/会签/或签/比例完成 -> 审批组终态。

失败闭环：无候选人、重复成员、失效员工、自审冲突和组织网关异常均形成结构化诊断；运行中组织变化不重写候选快照；重复完成不二次推进。

完成定义：候选和审批组语义独立于 Flowable multi-instance；非候选人不能处理任务；所有成员动作与组完成条件可审计。

### 12.3 M3 审批对象与数据治理

业务线：登记业务契约 -> 提交业务对象 -> 校验并生成四类数据 -> 按节点裁剪详情 -> 版本化修改 -> 终态冻结与结果输出。

状态：已于 2026-07-13 完成实现与真实运行验收并关闭，证据见 `docs/superpowers/specs/2026-07-13-bpm-m3-approval-data-governance-acceptance.md`。

失败闭环：敏感字段越权、业务版本冲突、非法字段修改和来源系统故障均失败关闭；审批快照仍可阅读；数据变化按契约锁定、展示差异或重新发起。

完成定义：一个内置通用申请证明审批人能看到标题、摘要、明细和附件；路由事实与工作数据分离；动作证据不可覆盖；最终结果使用冻结版本。

### 12.4 M4 核心审批运行与工作台

状态：已于 2026-07-13 完成实现与真实运行验收并关闭，证据见 `docs/superpowers/specs/2026-07-13-bpm-m4-core-runtime-workbench-acceptance.md`。M1-M4 首个产品基线同时关闭。

业务线：发起 -> 匹配定义 -> 创建实例 -> 计算路由 -> 生成任务/抄送 -> 读取审批详情 -> 执行动作 -> 退回重提/撤回取消 -> 终态与结果事件。

失败闭环：重复请求、过期任务版本、越权动作、任务已完成、定义失效和数据版本冲突均被拒绝并审计；失败命令不产生第二次有效推进。

完成定义：内置通用申请完成一条条件与多人审批主路径及关键异常路径；待办、已办、抄送、实例详情和 authored 运行图事实一致；M1-M4 首个产品基线关闭。

### 12.5 M5 高级流程运行

状态：已于 2026-07-13 完成实现与真实运行验收并关闭，当前 `RELEASABLE`，证据见 `docs/superpowers/specs/2026-07-13-bpm-m5-advanced-runtime-acceptance.md`。

业务线：配置 SLA/延迟/登记连接器/外部等待/子流程 -> 发布冻结 -> 创建计时器或等待事实 -> 事件/回调/子流程推进 -> 超时、失败恢复或补偿。

失败闭环：重复与乱序回调、验签失败、连接器网络失败、服务重启、计时器和人工动作竞态、子流程失败及父取消均有幂等、重试、暂停、补偿或人工处置路径。

完成定义：真实 timer、成功回调、重复回调、重启恢复和父子流程传播均有运行证据，不产生双任务、双推进或双终态。

### 12.6 M6 配置化业务接入

业务线：登记来源系统 -> 发布业务契约 -> 配置详情布局和流程绑定 -> 发起审批 -> 内外入口读取同一任务 -> 统一动作命令 -> 可靠交付结果。

失败闭环：未授权应用、未映射员工、绑定冲突、签名失败、重复请求、接收端故障和事件版本冲突均有拒绝、重试、死信和人工补偿记录。

完成定义：外部采购模拟器仅通过配置和固定协议接入，不新增采购专用 BPM handler；同一任务可在 Hunyuan 和外部入口安全处理；结果至少一次交付但只形成一个业务终态。

### 12.7 M7 运营与治理

业务线：监测积压/SLA/失败事件 -> 定位定义、实例、节点和外部记录 -> 授权处置 -> 重试/补偿/终止 -> 复核结果 -> 审计和报表。

失败闭环：所有处置要求原因、权限、幂等命令和前后快照；高风险动作可配置双人复核；运营页面不允许直接修改 Flowable 或业务表。

完成定义：运营人员能从告警定位根因并完成一次可审计恢复；报表只读取运行投影与审计事实；敏感字段按业务契约脱敏。

### 12.8 M8 定义演进与受限迁移

业务线：发布 V2 -> Graph 语义 Diff -> 影响分析 -> 实例分类 -> 映射和预演 -> 人工确认 -> 迁移/继续旧版/补偿重启 -> 审计。

失败闭环：不满足安全条件、映射不完整、存在活动任务或外部副作用时明确阻断；迁移命令幂等；执行失败不篡改原实例历史。

完成定义：V2 默认只影响新实例；合格静止实例可预演并迁移；不合格实例明确继续旧版或进入人工处置，不提供任意回滚承诺。

## 13. 空库重建与现有代码处理

数据库清空意味着不需要迁移旧流程定义和实例，但不代表删除所有已验证知识。

可以吸收的成果：

- 受控 BPMN 片段编译、路由 delegate、稳定 compiled ID 和节点快照思路。
- 候选解析、顺序审批、并行审批组及任务授权语义。
- 字段权限、数据版本、变更账本、退回重提和终态冻结语义。
- SLA、计时器、连接器安全、外部等待、幂等、重试和恢复状态机。
- 已有自动化与运行验收中的业务断言。

必须由新架构替换的边界：

- 旧作者 JSON、树形设计器、递归深度模型和旧模型导入兼容。
- 直接解析旧 JSON 的校验、候选预检和发布旁路。
- 以旧编译器类作为新平台入口。
- 为旧字段保留的模型、定义和快照表结构。
- 业务专用 handler 作为标准接入主路径。

复用必须通过新契约重新证明。旧测试通过只能说明算法资产存在，不能代替新 Graph、发布、运行和业务接入验收。

## 14. 总体验收矩阵

| 场景 | 必须证明 |
| --- | --- |
| 流程配置 | 管理员无需改代码即可配置审批、条件、并行、抄送、字段权限和时限并发布。 |
| 内置通用申请 | 申请人发起、领导查看完整审批对象、多人处理、退回/取消和终态全部闭环。 |
| 外部采购接入 | 只通过系统、契约、布局、绑定和事件配置接入；内外入口处理同一任务。 |
| 数据变化 | 金额或明细变化按策略锁定、展示差异或重新发起，旧证据不被覆盖。 |
| 身份与安全 | 未映射员工、伪造应用、越权字段、过期任务和重复请求均被拒绝并审计。 |
| 时间与外部故障 | timer、回调、连接器、服务重启和接收端故障可恢复，不重复推进。 |
| 子流程 | 父子版本、输入输出、完成、拒绝、取消和失败传播均可解释。 |
| 运营处置 | 异常可定位、可授权重试/补偿/终止，不能依赖数据库人工修复。 |
| 流程升级 | 新实例使用 V2，旧实例继续 V1；只允许满足条件的实例预演迁移。 |

## 15. 平台完成定义与文档治理

平台达到本蓝图最低目标必须同时满足：

1. 流程可由管理员完整配置、校验、模拟和发布。
2. 审批人能看到稳定、授权且可追溯的业务依据。
3. 内外入口共享同一任务权限、版本、幂等和审计边界。
4. 定义、业务配置、编译产物和运行事实均可解释并冻结版本。
5. 跨系统失败、数据变化、异常处置和流程升级都有明确闭环。
6. M1-M8 的完成状态均由对应业务主路径、关键异常路径、权限边界和真实运行证据证明。
