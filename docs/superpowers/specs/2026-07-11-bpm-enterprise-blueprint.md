# Hunyuan 企业级流程引擎总体蓝图

- 日期：2026-07-11
- 状态：总体设计基线

- 后端参考线：`E:\my-project\huanyuan-pro-jichu\ruoyi-vue-pro-master`

## 1. 结论

Hunyuan BPM 当前已经形成企业审批运行时的主体：定义治理、表单、线性审批、顺序多人、受限并行全员会签、员工任务动作、抄送、通知、追踪、可靠回调和审批数据治理。它下一阶段的主要问题不再是补任务按钮，而是建立能够持续扩展的流程建模、编译、时间事件、流程组合和运营治理平台。

总体目标不是复制 Yudao/RuoYi，而是建设一个以 Flowable `7.2.0` 为内部执行器、以 Hunyuan 领域模型为外部契约的企业级流程引擎：

```text
业务人员使用 Hunyuan 设计器表达流程
-> Hunyuan 校验并冻结定义
-> Hunyuan 编译为受控 BPMN
-> Flowable 执行内部令牌流转
-> Hunyuan 投影任务、审批组、路由、时间和事件事实
-> 员工、管理员和业务模块只使用 Hunyuan API
```

## 2. 设计目标

1. 从“线性 userTask 编译器”演进为“受控流程片段编译平台”。
2. 支持企业审批最常用的条件、分支、多人策略、时间、事件和子流程语义。
3. 保留已经验证有效的定义快照、候选预检、审批组、任务投影、数据版本、可靠回调和人工补偿机制。
4. 所有公共行为都能通过结构化 Hunyuan 契约解释，不要求前端解析 BPMN XML、Flowable ID 或内部 JSON。
5. 每个模块都能以真实业务流闭环，不以文件存在、页面出现或测试启动作为完成。
6. 建立稳定路线图，后续需求先映射到模块，不再完成一个小点后重新寻找方向。

## 3. 非目标

- 不把 Hunyuan 建设成任意脚本可执行的通用工作流 IDE。
- 不直接迁移参考项目的包结构、数字枚举、数据库表、接口名称或前端组件。
- 不把 Flowable RuntimeService、Task、Execution、Gateway ID 暴露给业务模块或前端。
- 不一次性支持全部 BPMN 2.0 元素。
- 不以 Flowable multi-instance 作为所有多人审批的默认答案。
- 不允许普通设计者配置任意 URL、任意 EL、任意 Java Bean 或任意脚本。
- 不推倒已完成的 Hunyuan 原生运行时；只有证据表明当前结构阻碍新模块时才局部重构。

## 5. 总体架构

### 5.1 六层结构

```text
设计体验层
  模型设计器、表单设计器、策略配置、发布检查

定义治理层
  AST/DSL、schema、静态校验、候选预检、版本、快照

编译部署层
  片段编译器、ID 命名、BPMN 生成、编译快照、部署认领

运行时领域层
  实例、任务、审批组、路由决定、计时器、外部等待、父子流程

可靠性与运营层
  命令、通知、回调、重试、补偿、异常事件、SLA、报表、审计

业务接入层
  发起契约、业务键、结果事件、连接器、幂等、样板业务
```

### 5.2 Flowable 边界

Flowable 负责：

- BPMN 部署。
- 令牌与执行树推进。
- user task、gateway、timer、receive task、call activity 等内部执行。
- 引擎事务内的任务和事件回调。

Hunyuan 负责：

- 业务可理解的模型、规则和安全约束。
- 定义发布、版本、权限和候选人预检。
- 任务、审批组、路由、计时器、外部等待和父子流程投影。
- 数据版本、审计、业务事件、重试和人工补偿。
- 员工端、管理端和业务模块公共 API。

## 7. 建设阶段

### 阶段 0：确认数据治理基线

仓库验收记录已证明审批数据治理闭环。进入 M1 实施前复核用户回归测试没有阻断问题；发现问题时在 M3 所有权内定向修复，不重新设计数据基座，也不让 M1 并行修改相同运行时区域。

### 阶段 1：建立非线性编排核心

M1 已使模型从顺序数组演进为受控 AST，支持排他、并行、包容分支及固定汇合，并将编译器重构为片段组合器。definition `21` 的四条数据库实流与 Chrome route/copy 证据均已通过，本阶段关闭；实现证据见 `docs/superpowers/specs/2026-07-11-bpm-m1-modeling-compiler-acceptance.md`。

### 阶段 2：建设时间和事件运行时

完成 M4，增加任务 SLA、超时动作、延迟节点、受控连接器和外部回调等待。流程从只由人工按钮驱动演进为可由时间和外部事件驱动。

### 阶段 3：统一审批策略和组织来源

完成 M2，将现有审批组扩展为明确的成员解析、完成策略、终止策略和空集合策略；再引入或签、比例审批、岗位、用户组等能力。

### 阶段 4：建设流程组合能力

完成 M5，以稳定的定义版本引用、变量映射、父子实例投影和取消传播实现子流程。多子流程实例建立在统一完成策略之上。

### 阶段 5：关闭运营和开放平台

完成 M6、M7，补齐流程运行图、SLA、异常处置、报表、打印、附件、评论、接入契约和至少两个不同业务模块的真实复用证明。

## 8. 统一设计原则

### 8.1 冻结而非回读

运行实例只读取发布定义和运行事实快照，不读取模型草稿、后来修改的表单或后来变化的组织配置来解释历史。

### 8.2 受控表达而非任意执行

条件使用类型化操作符和白名单字段；连接器使用登记配置和凭据引用；监听器使用登记键。用户输入不得直接成为 Flowable EL、URL、Bean 名或脚本。

### 8.3 结构化投影而非名称推断

任务组、路由决定、计时器、外部等待、父子流程关系都有 Hunyuan 结构化投影。页面不得从 task key 后缀、名称、BPMN XML 或 JSON 诊断字段推断业务状态。

### 8.4 一个动作一个事实

审批、路由、超时、回调、取消和子流程传播必须拥有幂等键、稳定终态和可审计事实；网络重试不能产生第二次有效推进。

### 8.5 默认失败关闭

权限快照、路由规则、连接器配置、子流程引用或数据版本损坏时，不猜测、不静默跳过，停止相关推进并进入可诊断状态。

### 8.6 业务单据拥有业务生命周期

BPM 管理审批过程，不直接成为采购、费用、请假等业务数据主表。业务模块通过稳定接入契约发起、查询和消费结果。

## 9. 参考项目借鉴规则

每项参考能力必须作出一种决策：

| 决策 | 含义 |
| --- | --- |
| `KEEP` | Hunyuan 当前机制合理，继续保留 |
| `HARDEN` | 当前方向正确，需要增强边界和验收 |
| `REFACTOR` | 当前结构阻碍后续，需要定向重构 |
| `ADAPT` | 借鉴参考机制，但按 Hunyuan 重新设计 |
| `DEFER` | 企业级有价值，但不属于当前阶段 |
| `REJECT` | 风险或边界不适合 Hunyuan，不纳入目标 |

不允许使用“参考项目存在”作为采用理由。必须回答它解决什么问题、依赖什么假设、Hunyuan 是否有同样问题、如何验证采用后的行为。

## 10. 跨模块契约

- 模型中的字段引用统一使用发布表单 schema 的 `fieldKey`。
- 运行时数据统一使用实例 `formDataVersion` 和冻结快照。
- 所有可驱动流程的入口统一进入命令边界，不直接散落调用 Flowable。
- 所有异步动作统一具有业务幂等键、执行记录、重试和人工处置状态。
- 所有生成节点保存 authored 身份和 compiled 身份映射。
- 所有运行事实进入统一 trace 时间线，但保留各领域结构化详情。
- 管理端完整视图和员工授权视图保持独立查询边界。

## 11. 总体验收

企业级蓝图不是在所有 BPMN 元素完成时关闭，而是在以下业务证明成立时达到当前目标：

1. 设计者能配置包含条件、并行/包容、审批、延迟、外部等待和子流程的受控流程。
2. 发布检查能阻断结构、人员、字段、连接器和版本错误。
3. 同一业务单据能按数据选择不同路径，并保留可解释路由事实。
4. 任务超时、延迟和外部回调能可靠推进，重复事件不重复流转。
5. 多人审批、退回重提、撤回取消和子流程传播具有稳定状态机。
6. 员工和管理员能看懂当前路径、等待原因、处理人、SLA 和历史动作。
7. 失败命令、通知、回调、计时器和外部等待可重试、可补偿、可审计。
8. 至少两个不同业务模块通过同一接入契约完成真实闭环。
9. 旧线性定义和历史实例保持兼容，升级不重写历史事实。
