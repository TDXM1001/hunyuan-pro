# BPM 审批规则与业务对象可视化配置设计

- 日期：2026-07-14
- 分支：`main`
- 状态：设计已确认，尚未实施
- 适用基线：`BPM-M1-M8-R1` 及后续兼容版本
- 设计范围：审批规则、业务对象、Graph 引用、双模式权限、版本兼容与验收

## 1. 背景与问题

当前“审批策略目录”和“业务契约目录”要求配置人员直接编辑策略 JSON 或契约 JSON。底层协议具有类型受控、版本不可变、发布冻结和摘要校验等正确工程属性，但当前页面把后端协议直接暴露给业务用户，导致以下问题：

1. 业务用户必须理解 `resolverType`、`resolutionPhase`、`changePolicy` 等内部字段。
2. 员工、角色、部门使用裸 ID，用户无法判断实际对象，环境迁移后也容易误配。
3. 校验结果只能说明 JSON 是否合法，不能说明实际会找到谁审批、哪些字段可修改。
4. 当前大弹窗无法承载动态分区、字段表格、版本引用、模拟和技术诊断。
5. 普通业务详情和技术协议未分层，权限边界与用户体验均不清晰。
6. 业务契约缺少字段名称、控件、提示、选项、明细结构和完整附件规则，无法成为成熟的业务对象设计器。

本设计保留现有后端策略与契约的规范化、版本化、冻结和失败关闭语义，将配置入口改造为面向业务管理员的可视化配置中心，同时为技术管理员保留独立授权的只读协议和诊断视图。

## 2. 目标与非目标

### 2.1 目标

1. 业务管理员不接触 JSON、内部枚举、数据库 ID 或正则表达式即可完成配置。
2. 技术管理员通过独立权限查看只读 canonical JSON、Schema 版本、摘要和诊断。
3. 审批规则和业务对象使用独立列表、编辑草稿和只读详情页面。
4. Graph 设计器只引用已启用版本，并展示名称、版本和自然语言摘要。
5. 后端接收结构化配置，负责语义校验、风险计算、规范化和 canonical 协议生成。
6. 现有 v1、已发布 Graph 和运行实例保持兼容，不做原地改写。
7. 不新增前端运行依赖或第二套 UI 框架。

### 2.2 非目标

1. 不新增 Graph 画布或替换现有 Graph 设计器。
2. 不提供任意表达式、脚本、SQL 或自定义代码插件。
3. 不建设通用低代码平台。
4. 不允许任何角色直接修改已启用版本的 canonical JSON。
5. 不自动迁移运行中实例或已发布 Graph 的冻结依赖。
6. 不引入新的表单设计器、状态管理库、拖拽库或 JSON 编辑器。

## 3. 用户模型与模式边界

系统采用双模式，但只维护一套后端语义和一份 canonical 协议。

### 3.1 业务管理员

业务管理员可以：

- 使用中文表单配置规则和业务对象。
- 通过员工、角色、部门、岗位和用户组选择器引用组织对象。
- 查看自然语言摘要、风险原因、规则模拟和申请预览。
- 保存、复制、校验、启用和退休有权限的版本。

业务管理员不能：

- 查看或编辑 canonical JSON。
- 输入裸员工、角色或部门 ID。
- 降低后端计算的风险等级。
- 修改已启用或已退休版本。
- 绕过服务端校验、模拟或高风险启用权限。

### 3.2 技术管理员

技术管理员在业务视图之外可以查看：

- Schema 版本。
- canonical JSON。
- SHA-256 内容摘要。
- 内部枚举和稳定字段 ID。
- 字段级规范化诊断。
- 版本差异和 Graph 冻结引用。

技术信息始终只读。修改必须复制为新草稿并回到结构化编辑器。

## 4. 信息架构与路由

正式事实源采用独立配置中心，Graph 只负责引用。

### 4.1 审批规则

| 页面 | 职责 |
| --- | --- |
| 规则列表 | 查询类型、名称、状态、版本、风险和引用数量 |
| 规则编辑 | 新建或编辑草稿、校验、模拟和启用 |
| 规则详情 | 只读摘要、配置、版本、引用、审计和技术信息 |

### 4.2 业务对象

| 页面 | 职责 |
| --- | --- |
| 对象列表 | 查询名称、来源系统、业务类型、状态和版本 |
| 对象编辑 | 配置业务编号、字段、明细、附件、修改规则和预览 |
| 对象详情 | 只读字段结构、版本、引用、差异和技术信息 |

### 4.3 页面承载原则

- 列表页继续使用现有列表页模式。
- 完整配置使用独立路由页面，不使用弹窗或抽屉。
- 单字段新增、复制确认、退休确认和高风险确认可使用短抽屉或弹窗。
- 现有 API 路径、菜单 ID 和角色授权优先兼容，避免破坏历史链接。
- 菜单显示名称建议将“审批策略目录”调整为“审批规则”，将“业务契约目录”调整为“业务对象”。

## 5. 审批规则编辑器

审批规则编辑页使用“基本信息、业务规则、异常处理、预览与测试”分区，不显示 JSON。

### 5.1 公共基本信息

- 规则名称。
- 规则编码，系统生成建议值并校验唯一性。
- 规则类型：发起范围、审批人、审批方式。
- 使用说明。
- 生命周期与版本。
- 后端计算的风险等级和风险原因。

### 5.2 发起范围规则

发起范围回答“谁能发起、谁能看到”。

发起人范围支持：

- 所有员工。
- 指定员工。
- 指定角色。
- 指定部门。
- 任意条件满足。
- 全部条件满足。

可见范围支持：

- 仅发起人。
- 发起人和流程参与人。
- 指定员工、角色或部门。
- 全部员工。

页面生成示例摘要：

> 管理员角色或财务部员工可以发起；发起人、流程参与人和财务部负责人可以查看。

### 5.3 审批人规则

审批人规则回答“任务到达后交给谁”。

审批人来源支持：

- 指定员工。
- 指定角色。
- 指定部门负责人。
- 发起人本人。
- 发起人的部门负责人。
- 发起人的直属上级。
- 指定岗位。
- 用户组。
- 管理层级链。
- 从流程判断字段读取员工。

异常处理必须显式配置：

- 找不到审批人：阻断流程、转指定人员、转指定角色或自动拒绝。
- 发起人也是审批人：允许、跳过本人、转部门负责人或阻断。
- 同一员工通过多个来源匹配时由系统去重。
- 多人顺序使用业务可理解的选择顺序或组织顺序，不暴露内部排序枚举。

普通模式默认在任务到达时解析审批人。发布时或发起时解析只在高级业务设置中显示，不进入技术 JSON 编辑。

### 5.4 审批方式规则

审批方式回答“几个人处理后才算完成”。

通过方式支持：

- 单人审批。
- 依次审批。
- 全部通过。
- 任意一人通过。
- 达到指定比例。

拒绝方式支持：

- 任意一人拒绝立即结束。
- 已经无法达到通过条件时结束。

允许动作支持通过、拒绝和退回。退回目标支持发起人、指定上级节点或按拒绝结束。多人审批达到终态后由系统关闭剩余待办，该内部终止细节不作为普通配置项。

### 5.5 自然语言摘要

前端根据结构化编辑模型即时生成预览摘要，后端在校验和保存时返回权威摘要。详情、列表、Graph 选择器和审计统一使用后端摘要。

示例：

> 任务到达时，从“财务经理”角色中查找审批人。发起人不能自审，改由其部门负责人处理；无人可处理时阻断流程。

### 5.6 规则模拟

业务管理员可以选择模拟发起人和可选路由字段。后端使用真实组织身份网关和候选解析器返回：

- 实际候选人及来源。
- 去重结果。
- 自审处理。
- 兜底处理。
- 阻断原因。
- 权威摘要。

模拟不创建实例、任务或业务事实，也不修改草稿。

## 6. 业务对象编辑器

业务对象是现有业务契约面向业务用户的表达。新建对象默认使用契约 Schema v2。

### 6.1 基本信息

- 业务对象名称。
- 对象编码。
- 来源系统，显示名称并保存稳定编码。
- 业务类型。
- 使用说明。
- 生命周期与版本。

### 6.2 业务编号规则

普通用户不填写正则表达式。页面提供：

- 任意非空编号。
- 固定前缀加流水号。
- 固定前缀加日期加流水号。
- 来源系统业务编号。
- 受控自定义格式。

页面实时展示有效示例和无效示例，后端生成并保存权威验证规则。技术详情可以查看最终正则。

### 6.3 申请内容字段

字段设计器使用可排序表格。字段配置包括：

- 字段名称和稳定编码。
- 数据类型。
- 输入控件。
- 是否必填。
- 数据等级。
- 提示文字和默认值。
- 显示顺序和布局宽度。

首批业务类型包括文本、多行文本、整数、金额、布尔、日期、日期时间、员工、部门、单选和多选。后端未支持的类型不得只在前端开放。

### 6.4 流程判断字段

现有 `routingFacts` 对业务用户显示为“流程判断字段”。字段配置包括名称、编码、类型、来源、数据等级、必填和是否可用于审批人解析。

值来源支持申请人填写、申请人组织信息、来源系统传入和其他申请字段映射。审批人规则引用时显示字段名称，不显示字段编码。

### 6.5 审批过程数据

现有 `workingDataSchema` 对业务用户显示为“审批过程数据”。字段可以配置：

- 发起时是否填写。
- 审批时是否可见。
- 哪些节点允许编辑。
- 终态是否冻结。
- 修改是否必须填写原因。

不同节点的字段权限需要契约 v2 与 Graph 节点权限共同实现，不能只保存前端配置。

### 6.6 明细与附件

契约 v2 增加正式 `lineItemSchema`，用于配置明细名称、行数限制、列字段、新增删除和排序规则。运行端不再要求业务用户填写明细 JSON。

附件规则支持是否允许、是否必传、最大数量、单文件大小、文件类型和审批人是否可补充。所有规则由提交接口服务端执行，不能只做前端校验。

### 6.7 数据修改规则

业务选项包括：

- 提交后不可修改。
- 每次修改生成新版本。
- 修改后必须重新发起。
- 只有指定审批字段可以修改。

选择指定字段时直接勾选审批过程数据。页面生成修改规则摘要，内部 `changePolicy` 枚举只在技术详情展示。

### 6.8 申请表单预览

编辑器提供桌面和移动预览，以及发起人、审批人只读和审批人编辑视角。预览复用现有运行表单渲染器和相同的契约到 form-create 规则转换，禁止维护第二套假预览。

## 7. 前端组件设计

### 7.1 复用组件

- 列表使用 `Page`、`ArtSearchPanel`、`ArtTable`、`ArtTableHeader` 和 `ArtTablePanel`。
- 编辑使用 `ArtEditPage`、`ArtEditSection`、`ElForm` 和 `ElFormItem`。
- 详情使用 `ArtDetailPage`、`ArtDetail`、`ElDescriptions`、`ElTimeline` 和 `ElTag`。
- 运行预览复用 form-create 运行渲染器。
- 技术 JSON 使用现有只读 JSON Viewer。

### 7.2 新增 BPM 业务组件

审批规则组件：

- `BpmRuleTypeSelector`
- `BpmScopeBuilder`
- `BpmIdentityPicker`
- `BpmCandidateRuleEditor`
- `BpmApprovalRuleEditor`
- `BpmRuleSummary`
- `BpmRuleSimulationPanel`

业务对象组件：

- `BpmBusinessKeyRuleEditor`
- `BpmSchemaFieldTable`
- `BpmSchemaFieldEditor`
- `BpmLineItemDesigner`
- `BpmAttachmentRuleEditor`
- `BpmChangePolicyEditor`
- `BpmApplicationPreview`
- `BpmTechnicalContractPanel`
- `BpmVersionReferencePanel`

身份选择组件在 BPM 范围内抽取员工、角色、部门、岗位和用户组选择器。只有出现第二个跨域真实消费者后，才提升为全局业务组件。

### 7.3 form-create 边界

form-create 只负责运行渲染和真实预览，不作为业务对象事实源。正式数据流为：

```text
业务对象字段设计器
  -> 受控契约模型
  -> 契约到 form-create 规则转换
  -> 运行表单与预览
```

### 7.4 依赖与排序

本设计新增前端运行依赖为 0。字段排序首版使用上移、下移按钮，不为拖拽排序新增依赖。

## 8. 后端模型与服务

### 8.1 单一事实源

前端提交结构化业务配置，后端执行结构校验、身份校验、业务语义校验、风险计算、规范化、摘要生成和 digest 计算，最终保存唯一 canonical JSON。不得同时保存可独立编辑的表单模型和 JSON 两套事实。

### 8.2 策略结构化请求

`PolicyDraftForm` 包含类型、编码、名称、说明、Schema 版本、CAS 修订号和类型化 `document`。`document` 分为发起范围、审批人和审批方式三个受控模型。

现有字符串 JSON 接口保留兼容读取边界，但不再作为业务编辑器主入口。业务接口不得信任前端计算的风险等级或摘要。

### 8.3 业务对象契约 v2

业务对象 v2 包含基本信息、业务编号规则、申请字段、流程判断字段、审批过程字段、明细、附件、修改规则和显示配置。

业务字段语义与显示语义分离：

- 业务语义保存 key、type、required 和 sensitivity。
- 显示语义保存 label、controlType、placeholder、helpText、order 和 span。

### 8.4 草稿保存与 CAS

新增正式草稿保存能力。请求携带精确版本和 `catalogRevision`。服务端锁定草稿、确认生命周期、校验修订号、保存规范协议并增加修订号。

已启用和已退休版本拒绝保存。过期修订返回 `CATALOG_REVISION_CONFLICT`，前端保留当前输入并允许查看服务器版本。

### 8.5 摘要、详情与技术接口分层

业务列表和详情只返回名称、类型、版本、生命周期、自然语言摘要、风险、引用数量和结构化业务配置。

canonical JSON、digest、内部枚举和规范化诊断只通过独立技术详情接口返回，并受独立权限保护。

### 8.6 结构化校验

校验结果统一包含：

- `pass`
- `riskLevel`
- `businessSummary`
- `findings[].code`
- `findings[].severity`
- `findings[].fieldPath`
- `findings[].message`
- `findings[].suggestion`

前端按 `fieldPath` 定位到具体编辑分区和控件。

### 8.7 风险计算

风险等级由后端根据语义计算。自动通过、自动拒绝、允许自审、全员发起、指定兜底和受限数据等因素形成风险原因。用户可以查看但不能降低等级。高风险启用继续使用独立权限、确认原因和追加审计。

## 9. 数据库与迁移

继续复用现有策略版本表、业务契约版本表、canonical JSON、digest、`catalog_revision` 和生命周期。

策略版本表补充可检索列：

- `policy_name`
- `description`
- `business_summary`
- `calculated_risk_level`

业务契约版本表补充：

- `object_name`
- `description`
- `business_summary`

新增版本化增量 SQL。展示列允许旧记录为空，旧记录名称回退到 key，摘要由后端根据 v1 动态生成。不得用 SQL 批量重写历史 canonical JSON。

## 10. 生命周期与版本兼容

### 10.1 草稿

- 可以保存、校验和模拟。
- 可以复制和删除未被引用的临时草稿。
- 不能被正式 Graph 发布引用。

### 10.2 已启用

- 业务配置只读。
- 可以查看引用、复制为新草稿和退休。
- 高风险启用需要独立确认。

### 10.3 已退休

- 只读且不能被新 Graph 引用。
- 已发布 Graph 和运行实例继续使用冻结版本。
- 可以复制为新草稿，不能恢复原版本为启用。

### 10.4 v1 到 v2

- v1 继续读取、发布冻结和运行。
- v1 详情提供中文只读摘要。
- “升级为可视化草稿”创建新的 v2 草稿，不修改 v1。
- 新 Graph 版本只能在 v2 启用后选择它。
- 运行实例不自动切换版本。

## 11. 权限设计

业务权限分别覆盖列表、业务详情、新建草稿、保存草稿、复制、校验、模拟、普通启用和退休。

技术权限独立覆盖：

- 查看 canonical JSON。
- 查看 digest 和内部诊断。
- 查看稳定字段 ID。
- 查看版本差异。
- 导出技术协议。

高风险启用继续使用独立权限。技术标签页隐藏不能替代服务端鉴权。

## 12. Graph 集成

Graph 设计器选择器只返回已启用规则和业务对象的名称、类型、版本、摘要、风险和阻断状态。

节点属性显示示例：

```text
审批人规则：费用审批人 v3
摘要：由财务经理审批；发起人不能自审

审批方式：全部通过 v2
摘要：全员通过；任一拒绝立即结束
```

发布仍冻结精确版本、canonical JSON、digest 和 Schema 版本。前端体验升级不得改变运行时不可变语义。

## 13. 异常处理

- 草稿冲突：保留用户输入，提供服务器版本查看和重新加载。
- 身份失效：显示失效对象名称和配置位置，阻止启用。
- 无人审批：显示模拟发起人、解析来源和阻断原因。
- 字段删除冲突：显示修改规则和 Graph 节点引用位置。
- 版本退休：显示已发布 Graph 引用数量和兼容影响。
- 高风险启用：显示风险原因，要求确认理由并审计。
- 技术权限不足：服务端拒绝，前端不渲染入口。

错误不得只返回内部错误码或 JSON 字段名，必须提供业务消息和可操作建议。

## 14. 响应式与可访问性

- 桌面端主编辑区占主要宽度，摘要或预览位于右侧。
- 宽度不足时摘要移动到编辑区下方。
- 移动端使用单列，表格只在自身容器滚动。
- 页面不得产生横向溢出或按钮覆盖。
- 技术 JSON 在移动端默认折叠。
- 字段排序和所有关键操作必须可通过键盘完成。

## 15. 测试设计

### 15.1 后端

- 三类结构化策略生成正确 canonical JSON，并能语义一致地反解析。
- 身份对象失效、自审、无人、去重和兜底行为正确。
- 模拟与真实运行使用相同解析器。
- 风险等级只由后端计算。
- 草稿 CAS 冲突不能覆盖。
- 非草稿不能保存。
- 技术接口权限隔离。
- 契约 v1 继续校验和冻结。
- 契约 v2 字段、显示、明细、附件和修改规则校验完整。
- v1 升级只创建 v2 草稿。
- Graph 冻结精确 v2 和 digest。
- 通用申请按 v2 校验真实数据。

### 15.2 前端

- 三类规则动态字段正确。
- 身份选择器显示名称并正确回显。
- 普通模式不存在可编辑 JSON。
- 摘要随配置更新，后端摘要覆盖前端预览。
- 校验错误定位到字段。
- 模拟展示候选人、来源和阻断。
- 冲突保留用户输入。
- 生命周期控制正确。
- 字段新增、编辑、删除、排序和编码冲突正确。
- 明细、附件和修改规则生成正确请求。
- 三种表单预览与运行端一致。
- 技术入口受权限控制。

### 15.3 浏览器闭环

业务管理员必须在不接触 JSON 的情况下：

1. 创建并启用发起范围、胡克审批人和全部通过三条规则。
2. 模拟解析得到胡克。
3. 创建并启用费用申请业务对象。
4. 配置金额、原因、审批备注、明细和附件规则。
5. 查看真实申请预览。
6. 在 Graph 中绑定中文名称和版本并发布。
7. 通过通用申请发起。
8. 胡克收到待办并审批。
9. 我的申请、待办、已办、实例和任务状态一致。

技术管理员必须能查看同一版本的 canonical JSON、digest、诊断和引用，但不能直接修改。普通业务账号直接请求技术接口必须被服务端拒绝。

## 16. 交付顺序

### 16.1 审批规则闭环

结构化 DTO、草稿保存、身份选择、规则编辑、摘要校验、模拟、技术详情和 Graph 引用一起交付。

### 16.2 业务对象闭环

契约 v2、字段设计、编号规则、明细附件、修改策略、真实预览和通用申请一起交付。

### 16.3 兼容与治理闭环

v1 摘要、v2 升级、引用关系、高风险权限、技术诊断和桌面移动验收一起交付。

每个闭环都跨前端、API、后端、数据库、测试和真实业务流完成，不交付只能单层演示的半成品。

## 17. 完成标准

1. 业务管理员完全不接触 JSON 即可配置三类审批规则。
2. 业务管理员通过字段设计器完成业务对象配置。
3. 员工、角色和部门均显示名称，不暴露裸 ID。
4. 每条规则具有中文摘要和真实模拟结果。
5. 申请预览和运行端使用同一转换逻辑。
6. 后端保存唯一 canonical 协议并冻结精确版本。
7. v1、已发布 Graph 和运行实例保持兼容。
8. 技术 JSON 由独立权限保护且只读。
9. 高风险由后端计算并受独立启用权限控制。
10. 自动化、真实 Graph 发布、发起、审批和终态链全部通过。

满足以上条件后，才可以把当前开发者协议页面视为已升级为可面向业务管理员交付的配置产品。
