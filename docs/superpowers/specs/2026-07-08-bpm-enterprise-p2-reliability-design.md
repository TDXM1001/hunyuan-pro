# Hunyuan BPM 企业级流程引擎 P2 可靠性设计

## 结论

P2 推荐采用 **可靠性内核优先 + 业务样板验收**。

当前 `hunyuan-bpm` 已经具备 P1 级别的 Hunyuan 原生流程平台基线：定义治理、运行工作台、管理员干预、业务接入契约、集成监控菜单、手工抄送、实例轨迹和高级审批动作已经有代码与验收记录。P2 不应继续零散追齐参考项目的功能目录，而应把流程平台升级到企业级最关键的可靠性层：关键事件可查、通知投递可查、业务回调可执行可重试、失败可解释可补偿。

一句话目标：任意一个流程实例出问题时，平台管理员能回答“发生了什么、通知了谁、业务侧收到没有、失败后怎么处理”。

## 当前基线

已具备能力：

- `hunyuan-bpm` 已作为独立后端模块接入 `hunyuan-admin`。
- Flowable 7.2.0 作为 BPM 内部隐藏内核，不直接暴露给公开 controller、VO、前端 API 类型。
- `t_bpm_callback_record` 与 `t_bpm_command_record` 已存在，具备回调状态、重试次数、下次重试时间、命令状态等基础字段。
- 管理员端已有集成监控入口：回调记录列表、命令记录列表。
- 业务接入侧已有 `BpmBusinessProcessApi`、`BpmBusinessStartCommand`、`BpmBusinessInstanceStatus`、`BpmBusinessResultEvent`。
- `hunyuan-base` 已有站内信、短信、邮件基础能力，BPM 不需要重造通用消息系统。

仍需补齐：

- 回调记录目前偏“可查 + 手动探针”，还没有形成稳定的执行、失败、重试、补偿闭环。
- BPM 事件还分散在实例、任务、动作日志、回调和命令记录中，缺少按实例追踪的一条线。
- 通知没有 BPM 级投递记录，无法从流程实例定位到通知渠道、接收人、发送结果和失败原因。
- 监控页面还偏列表查询，缺少面向运维排障的“实例追踪视角”。
- 业务样板还没有证明真实业务从发起到审批结果回写的闭环。

## 参考项目取舍

### 借鉴机制

从 `yudao-ui-admin-vue3-master` 借鉴：

- 流程详情中把表单、节点、审批记录、评论、附件、操作区组织成一个排障入口的思路。
- 流程中心对“我的待办、我的已办、我的抄送、实例管理、流程报表”的信息架构。
- 设计器中对通知、监听器、触发器、候选人规则的产品分组方式。

从 `ruoyi-vue-pro-master` 借鉴：

- 业务通过流程实例和业务标识建立关系，而不是业务模块直接读取 Flowable 对象。
- 事件、监听器、消息、HTTP 触发形成扩展点。
- 候选人解析、流程发布校验、任务动作集中在 BPM service 层。
- 失败记录、重试和可观测入口要成为平台能力，而不是散落在业务模块中。

### 不迁移内容

- 不迁移参考项目的 `/admin-api/bpm/**` 路径。
- 不迁移参考项目的 VO、DO、枚举、权限码、菜单结构。
- 不暴露 Flowable `Task`、`ProcessInstance`、`ProcessDefinition` 给 Hunyuan 公开 API。
- 不在 P2 做通用 HTTP 节点执行平台。
- 不在 P2 做复杂会签、子流程、边界事件、任意节点跳转或脚本执行平台。
- 不把短信、邮件、站内信能力复制进 BPM；BPM 只记录自己的触发和投递结果。

## P2 范围

### 1. BPM 事件记录

目标：形成面向实例追踪的统一事件流水。

建议新增或固化的事件类型：

- `INSTANCE_STARTED`
- `TASK_CREATED`
- `TASK_APPROVED`
- `TASK_REJECTED`
- `TASK_RETURNED`
- `TASK_TRANSFERRED`
- `TASK_DELEGATED`
- `TASK_ADD_SIGNED`
- `TASK_REDUCE_SIGNED`
- `TASK_RECALLED`
- `INSTANCE_CANCELLED`
- `INSTANCE_FINISHED`
- `COPY_SENT`
- `BUSINESS_CALLBACK_CREATED`
- `BUSINESS_CALLBACK_SUCCEEDED`
- `BUSINESS_CALLBACK_FAILED`
- `NOTIFICATION_CREATED`
- `NOTIFICATION_SUCCEEDED`
- `NOTIFICATION_FAILED`

事件记录至少应支持：

- 按 `instanceId` 查询完整生命周期。
- 关联 `taskId`、`businessType`、`businessId`、`eventKey`、`actorEmployeeId`。
- 保存事件摘要、原始 payload 摘要、失败原因、发生时间。
- 不保存大对象，不把表单全量快照重复存入事件表；表单快照仍由实例/任务表承载。

### 2. 通知投递记录

目标：BPM 不重造消息系统，但能解释自己触发了什么通知。

通知记录应覆盖：

- 实例发起后通知首个待办人。
- 新待办产生时通知当前处理人。
- 审批完成后通知发起人。
- 拒绝、退回、撤回、取消时通知相关人。
- 抄送产生时通知被抄送人。

每条 BPM 通知记录至少包含：

- `instanceId`
- `taskId`
- `channel`：站内信、短信、邮件。
- `receiverEmployeeId`
- `receiverSnapshot`
- `templateCode`
- `sendStatus`
- `requestPayloadJson`
- `responseSnapshotJson`
- `failReason`
- `retryCount`
- `sentAt`

实现原则：

- 站内信优先复用 `hunyuan-base` 的 `MessageService`。
- 短信优先复用 `SmsService` 与 `t_sms_send_log`。
- 邮件优先复用 `MailService`。
- BPM 通知记录只保存 BPM 上下文和投递结果，不替代通用消息日志。
- P2 先支持站内信完整闭环，短信和邮件可以先完成记录与可扩展接口，实际投递按当前基础能力成熟度推进。

### 3. 业务回调执行器

目标：把当前回调记录推进为可执行、可失败、可重试、可补偿的闭环。

回调生命周期：

1. BPM 产生业务结果事件。
2. 写入 `t_bpm_callback_record`，状态为待处理。
3. 执行器读取待处理或到期重试记录。
4. 调用业务回调处理器。
5. 成功则标记成功，写入响应摘要和完成时间。
6. 失败则记录失败原因、递增重试次数、计算 `nextRetryAt`。
7. 超过最大次数后标记为需人工补偿。
8. 管理员可手动重试或标记已补偿。

关键约束：

- 回调必须幂等：同一个 `eventId` 或业务幂等键重复执行时，不得重复推进业务状态。
- 回调执行失败不能回滚已经完成的 BPM 审批动作。
- 业务处理器异常必须被记录，不能只写日志。
- 手动重试和自动重试走同一套执行器，不各写一套逻辑。
- P2 可以先用应用内定时扫描，不新增消息队列依赖。

### 4. 命令记录增强

目标：业务发起和关键审批命令能被追踪、排错和解释。

增强点：

- 命令记录保存请求摘要、执行结果、失败原因、关联实例 ID。
- `START:{businessType}:{businessId}:{definitionKey}` 继续作为发起幂等键。
- 管理员监控页能按业务类型、业务 ID、实例 ID、命令状态筛选。
- 重复发起时应返回同一实例或明确拒绝，不能产生重复流程。

### 5. 实例追踪视角

目标：在管理员排障时，不需要分别打开实例、任务、回调、通知、命令多个页面猜。

P2 需要一个统一查询能力：

- 输入 `instanceId`。
- 返回实例基础信息。
- 返回当前任务和历史动作。
- 返回 BPM 事件流水。
- 返回业务命令记录。
- 返回业务回调记录。
- 返回通知投递记录。

前端可以先不做复杂大屏，优先在现有实例详情抽屉或管理员实例列表中增加“追踪/可靠性”页签。页面保持 Hunyuan 现有安静、密集、可扫描风格。

### 6. 业务样板验收

目标：用一个最小真实业务验证可靠性底座，而不是为了演示写一个孤立脚本。

推荐样板：

- 费用申请或采购申请。
- 业务侧生成申请单。
- 业务侧通过 `BpmBusinessProcessApi.start` 发起 BPM。
- BPM 审批通过或拒绝后生成业务结果事件。
- 回调执行器把结果写回样板业务状态。
- 失败时写入回调失败记录，可手动重试。

样板边界：

- 样板不进入 BPM 底座核心表。
- 样板不强制成为所有业务模块模板。
- 样板只验证接入契约、幂等、回调、补偿、监控。

## 数据流

### 正常审批闭环

1. 员工或业务模块发起流程。
2. BPM 写入实例、任务、命令记录和 `INSTANCE_STARTED` 事件。
3. BPM 产生待办并写入 `TASK_CREATED` 事件。
4. BPM 触发通知，写入通知记录。
5. 审批人处理任务。
6. BPM 写入动作日志与任务事件。
7. 实例结束时写入 `INSTANCE_FINISHED` 事件。
8. BPM 生成业务结果事件，写入回调记录。
9. 回调执行器调用业务处理器。
10. 回调成功后写入 `BUSINESS_CALLBACK_SUCCEEDED` 事件。

### 失败与重试闭环

1. 回调执行失败。
2. 回调记录保存失败原因和响应摘要。
3. 事件流水写入 `BUSINESS_CALLBACK_FAILED`。
4. 执行器计算下次重试时间。
5. 自动重试或管理员手动重试。
6. 重试成功则标记成功。
7. 达到最大重试次数仍失败则标记需人工补偿。
8. 管理员处理后可标记已补偿，并写入补偿事件。

## 错误处理

- BPM 审批动作成功后，通知或回调失败不应撤销审批动作。
- 同一个事件重复投递必须幂等。
- 回调处理器不可用时，失败记录必须可查，并能看到下一次重试时间。
- 通知渠道不可用时，站内信、短信、邮件的失败应分渠道记录。
- 管理员手动重试必须记录操作者、操作时间和结果。
- 人工补偿必须记录原因，不能静默改状态。

## 前端体验

优先补齐现有页面，不另起一套监控系统：

- 回调记录列表增加重试结果、下一次重试时间、最后失败原因、人工补偿状态。
- 命令记录列表增加实例跳转、业务标识筛选、失败原因。
- 实例详情抽屉增加“可靠性追踪”页签，展示事件、通知、回调、命令。
- 管理员从实例列表可进入同一追踪视角。
- 普通员工运行端不展示平台级失败细节，只展示必要的流程状态。

## 后端边界

- `hunyuan-bpm` 负责 BPM 事件、通知上下文、回调记录、命令记录、执行器和监控查询。
- `hunyuan-base` 负责站内信、短信、邮件等通用投递能力。
- 业务模块负责自己的业务状态变更和幂等处理。
- `hunyuan-admin` 只承载菜单、权限、组织员工上下文和后台入口。
- Flowable 仍只允许在 BPM 内部网关和适配层出现。

## 分阶段实施建议

### P2.1 事件流水与实例追踪

- 建立 BPM 事件记录模型。
- 在实例、任务、回调、通知关键点写入事件。
- 提供按实例查询追踪接口。
- 前端实例详情增加追踪页签。

验收：

- 任意实例能查到从发起到结束的关键事件。
- 事件记录不暴露 Flowable 原生对象。
- 后端单测覆盖发起、审批、取消、回调失败事件。

### P2.2 通知投递记录

- 定义 BPM 通知命令和通知记录。
- 先完成站内信闭环。
- 预留短信、邮件渠道，不复制通用投递逻辑。
- 前端展示通知记录与失败原因。

验收：

- 新待办产生后能查到通知记录。
- 站内信成功和失败都可追踪。
- 通知失败不影响审批动作提交。

### P2.3 回调执行器与重试

- 实现统一回调执行器。
- 自动扫描待处理和到期重试记录。
- 手动重试复用执行器。
- 支持最大重试次数和人工补偿状态。

验收：

- 回调成功能标记成功并写入事件。
- 回调失败能记录原因、递增重试次数、设置下次重试时间。
- 重复回调不重复推进业务状态。

### P2.4 业务样板闭环

- 建立最小费用/采购申请样板。
- 样板通过 BPM 契约发起流程。
- 审批结果通过回调执行器回写样板状态。
- 浏览器或集成测试证明完整业务闭环。

验收：

- 样板业务从发起到审批结果回写闭环通过。
- 制造一次回调失败后能在监控页看到失败并手动重试。
- 样板代码不污染 BPM 底座边界。

## 完成定义

P2 可收尾的标准：

- 任意实例能从管理员端查看可靠性追踪。
- BPM 关键事件形成统一流水。
- 通知投递记录可查，至少站内信闭环可验证。
- 业务回调支持待处理、成功、失败、重试、需补偿、已补偿状态。
- 手动重试与自动重试共用同一执行路径。
- 一个最小业务样板证明业务发起、审批、结果回写、失败重试闭环。
- 后端单测、前端契约测试、类型检查、至少一条浏览器业务验收记录通过。
- 公开 API 和前端类型不暴露 Flowable 原生对象。

## 理解校验

关键假设：

- P1 已经形成可运行的 Hunyuan 原生 BPM 基线，P2 应在其上增强可靠性，而不是重做模块结构。
- 企业级的第一优先级是可恢复、可解释、可审计，而不是节点类型数量。
- 参考项目的价值在机制和能力地图，不在直接迁移代码、接口和页面。
- 当前阶段可以接受应用内定时扫描作为回调执行器，不新增消息队列依赖。

已验证依据：

- 当前仓库已有 `t_bpm_callback_record`、`t_bpm_command_record`、集成监控页面与相关验收记录。
- 当前仓库已有 `BpmBusinessProcessApi` 业务接入契约。
- 当前仓库已有站内信、短信、邮件基础模块可作为通知投递底座。
- 当前仓库已有 Flowable 隔离测试，P2 必须继续维持该边界。

仍需实施时验证：

- 事件表还是复用现有动作日志扩展，需要在实施计划中结合查询复杂度和表职责细化。
- 通知记录与通用消息日志之间的关联字段需要结合 `hunyuan-base` 现有实体最终确认。
- 回调执行器的调度方式需要结合当前作业模块和部署方式确认。
- 样板业务放置位置需要在实施计划中选择最小污染路径。

人工必须重点审阅：

- 事件状态和回调状态是否足够覆盖真实运维场景。
- 通知记录是否会重复保存敏感信息。
- 业务样板是否越界成生产业务模板。
- 自动重试策略是否会给业务模块造成重复状态推进。
