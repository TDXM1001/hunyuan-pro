## 4. 跨模块数据契约

### M1 路由

- 路由只能读取已提交的 `currentFormDataSnapshotJson + formDataVersion`。
- 路由决定记录输入版本。
- 隐藏字段可以参与服务端路由，但员工 trace 不泄露值。

### M2 审批策略

- 表单选人字段沿用同一 schema 和值校验。
- 审批人修改表单后，后续成员读取新版本。
- 并行写策略必须由 M3 明确；当前 `parallelAll` 保持只读。

### M4 时间和事件

- 超时动作不得写入用户未授权字段。
- 外部回调修改表单必须提交登记映射、预期版本和幂等键。
- 延迟期间数据变化的允许范围由节点快照决定。

### M5 子流程

- 输入输出使用显式字段映射和类型校验。
- 子流程输出合并必须提交父实例预期版本。
- 多子流程并行输出默认禁止写同一父字段。

### M6/M7

- 员工视图继续按权限裁剪；管理端完整视图单独授权。
- 报表、打印和业务事件必须标明数据版本。

## 5. 权威数据边界

```text
业务单据数据       由业务模块拥有
审批过程表单快照   由 BPM 实例拥有
执行控制变量       由 Flowable 内部拥有
字段变更历史       由 BPM 审计拥有
最终业务结果       通过版本化事件交给业务模块
```

任何模块不得把 Flowable variables 重新提升为审批数据真相，也不得让业务模块绕过版本和审计直接修改 BPM 实例快照。

## 6. 需要重点审查的结构

1. `BpmInstanceService`、`BpmTaskService` 与 `BpmFormDataMutationService` 的事务边界是否真正原子。
2. 审批组、任务、实例锁顺序是否在所有动作路径一致。
3. 员工任务接口在快照损坏时是否严格不返回隐藏数据。
4. 回调记录是否冻结最终数据，而不是重试时重新读取实例。
5. 重提是否使用最新快照并保留旧引擎任务和变更历史。
6. 表单类型校验是否与设计器组件支持范围一致。

若测试发现这些结构有问题，应在 M3 内修正或局部重构，不延迟到后续模块绕开。

## 7. 兼容原则

- 旧定义无字段权限时默认只读。
- 无表单流程不要求版本和 patch。
- 历史实例版本为 1，不伪造历史变更。
- 已有单人、顺序和 `parallelAll` 动作语义不变。
- API 只增加可选字段。
- 后续 AST v2 继续复用同一字段权限对象，不引入第二套权限模型。
