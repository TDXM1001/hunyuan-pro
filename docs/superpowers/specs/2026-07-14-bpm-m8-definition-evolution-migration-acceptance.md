## 1. 当前能力事实

- Graph Diff 按稳定 node、edge 和 scope ID 比较语义、连接、策略、依赖与布局；JSON 属性顺序不产生伪变化，纯布局变化不建议迁移。
- 新版本默认只影响新实例；运行实例迁移要求同一流程键、严格前向版本、目标版本激活且已经部署，并在执行事务内锁定目标版本复核。
- 预演和执行同时检查活动人工任务、并行 token、timer、外部等待、子流程、不可逆回调副作用、节点映射、冻结业务契约、表单、工作数据和流程变量。
- 表单链式重命名始终从不可变源快照计算一次，Flowable 和平台投影使用同一迁移结果；工作数据追加新版本，不覆盖历史事实。
- 迁移通过 Flowable 官方 `ProcessMigrationService` 执行；批次使用执行租约、续租、owner fencing 和实例级新事务，部分失败保留逐项结果。
- 幂等键绑定规范化请求 SHA-256；唯一键竞争后使用 `FOR UPDATE` 当前读加载赢家，不在 MySQL `REPEATABLE READ` 快照中误判。
- 查询、预演、执行和审计使用独立权限与响应 VO。预演/执行不返回源目标快照；审计记录标识、摘要、变量映射名称、操作者、引擎观测和处置结果，不保存变量明文。
- 失败实例只能复核重试、保留源版本或登记真实补偿结果；已完成副作用不承诺物理回滚。

## 5. 完成定义对账

1. V2 发布不修改 V1 在途实例；稳定 ID Diff 能区分语义变化与纯布局变化。
2. 合格静止实例可完成 Diff、预演、确认、真实 Flowable 迁移并继续运行。
3. 活动任务、并行 token、timer、外部等待、子流程、副作用和不完整数据/变量映射均会明确阻断。
4. 请求哈希、批次/实例幂等键、数据库唯一约束、当前读竞争恢复和 owner fencing 防止重复迁移；部分失败保留逐项结果并可处置。
5. 任一迁移可追溯源/目标版本、映射、前后摘要、引擎观测、操作者与补偿结果，预演和执行响应不泄漏审计快照。
