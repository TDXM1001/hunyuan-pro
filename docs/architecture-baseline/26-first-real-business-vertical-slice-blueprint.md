# 首个真实业务纵切蓝图

## 1. 当前状态

截至 2026-07-24，平台架构迁移和既有能力处置已完成阶段性收口，但产品服务的真实用户、业务目标和核心业务对象仍未确定。因此，下一阶段不再新增平台迁移编号，也不从仓库中的历史模块名称反推产品需求。

本蓝图是首个真实业务纵切的准入模板。只有底座结构收口完成，并且第 3 节的业务契约得到真实业务方确认后，才能填写具体模块名称、创建数据库迁移或进入功能编码。底座结构的关闭范围和完成定义见 [27-foundation-structure-closeout-blueprint.md](27-foundation-structure-closeout-blueprint.md)。

当前结论：`BLOCKED_BY_FOUNDATION_STRUCTURE_CLOSEOUT`。

## 2. 选题原则

首个业务纵切必须同时满足以下条件：

1. 有明确的目标用户和实际工作场景，不以技术演示或历史示例作为需求来源。
2. 覆盖一个可以独立验收的端到端任务，而不是只有增删改查页面。
3. 至少包含一个归属清晰的核心业务对象及其状态变化。
4. 能明确区分操作权限、数据可见范围和业务状态约束。
5. 范围足够小，可以一次贯通数据库、后端、接口、前端和运行态验收。
6. 不要求提前引入微服务、消息队列、多数据库或新的终端形态。

组织目录、员工账号、角色权限、岗位目录和平台支持属于已经建立的平台能力。它们可以被真实业务调用，但不能单独作为本轮业务选题。

## 3. 业务契约

进入详细设计前，必须用可验证的业务语言填写以下内容。任何一项仍为“待定”时，不启动实现。

| 契约项 | 必须回答的问题 | 通过标准 |
| --- | --- | --- |
| 业务目标 | 要解决什么实际问题，当前成本或风险是什么 | 能用一句话描述结果，而不是描述系统功能 |
| 目标用户 | 谁发起、谁处理、谁审核、谁查看 | 每个参与角色都有真实职责和使用场景 |
| 核心任务 | 用户从什么触发点开始，要完成什么结果 | 有明确起点、终点和失败分支 |
| 端到端流程 | 任务经过哪些步骤、参与者和状态 | 主流程与关键异常流程均可画出 |
| 核心对象 | 哪个业务对象承载流程和数据 | 对象有唯一所有模块，关键字段来源明确 |
| 生命周期 | 对象有哪些状态，哪些动作触发状态变化 | 状态转换、前置条件和禁止操作完整 |
| 权限边界 | 谁能执行哪些动作、查看哪些数据 | 能映射为能力码、数据范围和后端校验 |
| 一致性边界 | 哪些写操作必须同时成功或失败 | 本地事务范围明确，失败补偿可说明 |
| 外部依赖 | 是否依赖外部系统、文件、通知或定时任务 | 输入输出、失败策略和责任方明确 |
| 验收指标 | 怎样证明业务结果有效 | 有可观察结果、测试样本和验收角色 |

## 4. 推荐纵切范围

首轮只选择一个高频主任务，建议控制为：

- 一个核心业务对象。
- 一条主流程和必要的异常分支。
- 两到四个真实参与角色。
- 一组稳定能力码和数据范围规则。
- 一个管理后台 feature 包。
- 一套可在浏览器中完成的验收样本。

以下内容默认不进入首轮：统计大屏、复杂报表、移动端、多租户、跨系统批量同步、通用流程设计器和为了未来可能性建设的扩展点。

## 5. 实施闭环

业务契约确认后，按以下顺序实施，不允许只交付其中一层：

1. **数据所有权与迁移**：明确表归属、约束、索引、初始化数据和 Flyway 迁移；使用 `utf8mb4`，禁止 ORM 自动修改生产结构。
2. **后端业务用例**：建立应用用例、领域规则和公开 Facade；跨模块只调用公开接口，不访问其他模块的 Mapper、DAO 或 Entity。
3. **公共接口契约**：接口进入 `/api/admin/v1`，同步生成并核验 OpenAPI；错误模型、幂等和并发行为必须可测试。
4. **前端业务 feature**：页面、状态、接口适配和测试归入独立业务 feature；主应用只负责装配、路由和权限入口。
5. **安全闭环**：后端校验身份、模块启用、能力码、数据范围和业务状态；前端可见性不能替代后端授权。
6. **运行态验收**：使用真实 MySQL 迁移、直接 API 和浏览器覆盖主流程、越权、非法状态、异常和恢复路径。
7. **代码图复核**：使用新的 Codebase Memory 项目名重建索引，检查模块依赖、路由、消费者和跨边界调用，再与源码、数据库和运行态交叉核对。

## 6. 完成定义

首个真实业务纵切只有同时满足以下条件才算关闭：

1. 业务契约已由真实需求输入确认，文档中不存在影响实现的待定项。
2. 数据表、接口、能力码、菜单和前端入口具有一致且唯一的业务语义。
3. 正常流程、异常流程、越权访问和非法状态均有自动测试。
4. Flyway 可在隔离 MySQL 重放，开发库迁移前有可校验备份。
5. OpenAPI、TypeScript 检查、前后端测试、生产构建和严格 UTF-8 校验通过。
6. 至少使用管理员和一个受限业务角色完成浏览器运行态验收。
7. Codebase Memory 未发现绕过公开边界的新增依赖，源码复核结果一致。
8. 验收证据、剩余限制和回滚边界写入独立关闭文档。
