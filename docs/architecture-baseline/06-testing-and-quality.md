# 测试与质量基线

## 1. 测试目标

测试的目的不是追求单一覆盖率数字，而是保护：

- 核心业务规则。
- 数据库约束和事务行为。
- 前后端接口契约。
- 登录、授权和数据权限。
- 用户关键业务路径。
- 模块边界和依赖方向。

测试范围随着业务风险增长，不要求所有简单代码都具有同等测试密度。

## 2. 后端测试分层

### 2.1 领域单元测试

适用于：

- 状态转换。
- 金额、时间和资格规则。
- 值对象和领域服务。
- 不依赖数据库的业务不变量。

这类测试应快速、稳定，不启动完整 Spring 容器。

### 2.2 应用用例测试

验证：

- 用例执行顺序。
- 模块公开接口调用。
- 权限、幂等和错误分支。
- 事务内需要共同成功的操作。

外部服务可使用测试替身，但不能因此忽略真实数据库行为。

### 2.3 MySQL 集成测试

使用 JUnit 5 和独立测试数据库验证。当前 Windows 开发环境使用 phpStudy 中与生产同系列的 MySQL，并通过独立数据库名、Redis DB 和显式环境变量与开发数据隔离：

- Flyway 可以从空库完整迁移。
- MyBatis/JPA 映射正确。
- 唯一约束、外键和索引符合预期。
- SQL 语法和排序规则符合 MySQL 行为。
- 事务、锁、并发和回滚行为正确。

不使用 H2 代替全部 MySQL 集成测试，因为两者的 SQL、类型、索引和锁行为存在差异。

集成测试默认不运行，只有 `HUNYUAN_IT_ENABLED=true` 且测试库名以 `_it` 结尾时才启用。测试代码不得回退连接开发库，也不得自动删除或清空非测试数据库。

### 2.4 API 测试

验证：

- 请求校验。
- 认证和授权。
- HTTP 状态与错误码。
- 分页、排序和过滤。
- 幂等行为。
- 不兼容契约变化。

## 3. 前端测试分层

### 3.1 单元与组件测试

使用 Vitest 验证：

- 无界面业务规则和格式转换。
- 模块注册和权限过滤。
- 关键组件状态。
- 表单规则和错误状态。

不要用大量脆弱快照替代行为断言。

### 3.2 浏览器流程测试

使用 Playwright 保护少量高价值流程：

```text
登录 -> 加载权限 -> 进入目标模块
创建或修改核心数据 -> 保存 -> 重新查询
执行关键业务动作 -> 状态和可用操作正确变化
无权限用户 -> 页面和接口均被拒绝
```

具体流程必须等真实业务确定后补充，不能将抽象示例直接当成验收用例。

## 4. 契约测试

前后端通过 OpenAPI 建立契约守卫：

- 生成文件必须可重复生成。
- 前端编译应能发现删除字段和类型变化。
- CI 应检测未更新的生成客户端。
- 不兼容 API 变化需要明确升级策略。

模块和能力编码也应有一致性检查，避免前端使用一个权限编码、后端校验另一个编码。

## 5. 架构测试

使用 Spring Modulith、ArchUnit 或等价工具检查：

```text
domain 不依赖 api
domain 不依赖 infrastructure 实现
interfaces 通过 application 进入业务
模块 A 不引用模块 B 的 Repository、Mapper 和内部 Entity
平台层不依赖具体业务模块
模块依赖不存在循环
```

架构规则必须自动执行，不能只依赖代码评审记忆。

## 6. 模块化测试矩阵

至少覆盖：

| 场景 | 预期 |
| --- | --- |
| 所有已实现模块启用 | 系统正常启动和运行 |
| 某可选模块关闭 | 其他模块仍可运行 |
| 模块依赖缺失 | 启动或装配阶段明确失败 |
| 用户缺少模块能力 | 前端无入口，后端拒绝请求 |
| 用户有页面能力但无操作能力 | 可以查看但不能执行目标动作 |
| 直接访问受限 URL 或 API | 无法绕过后端授权 |

## 7. 交付质量门

一次业务交付至少经过：

```text
编译与静态检查
  -> 单元测试
  -> MySQL 集成测试
  -> API/契约测试
  -> 关键浏览器流程
  -> 数据库迁移验证
  -> 人工业务验收
```

仓库测试通过只证明代码层验证完成，不等同于真实环境、真实角色和真实业务数据下的最终验收。
