# A3.4 OA 真实采用审计

## 1. 审计范围与结论

本审计批次沿用 2026-07-22 的 A3.4 执行批次，目标是判断 OA 是否仍有真实业务价值、
稳定消费者、有效数据和可保留的迁移边界。本批不直接退役 OA，也不修改历史基线迁移。

当前结论：**OA 不能整体退役，采用分域处置。**

| 范围 | 当前判定 | 结论依据 |
| --- | --- | --- |
| 通知、公告和员工通知查询 | `KEEP` | 后端仍有后台管理、员工查询、详情和查看记录接口；数据库仍有通知、通知类型和可见范围数据 |
| 企业主数据 | `ADAPT` | 后端接口、企业和员工关联数据、角色授权仍存在，但当前前端没有对应业务页面 |
| 企业员工关联 | `ADAPT` | 仍有真实企业员工关联记录，且依赖员工公开 Facade；需要先确定是否由组织/身份能力承接 |
| 银行信息 | `ADAPT` | 仍有有效数据和完整后端读写接口，但数据较旧且缺少当前前端入口 |
| 发票信息 | `ADAPT` | 仍有有效数据和完整后端读写接口，但数据较旧且缺少当前前端入口 |
| 数据库 OA 菜单及组件路径 | `ADAPT` | 角色仍被授权，菜单组件路径仍指向已不在当前源码中的页面；不能在替代入口和角色迁移前直接删除 |

因此，下一步不是删除 OA，而是完成 OA 的入口归属确认、数据责任确认和最小真实流程验收。

## 3. 前端入口与菜单消费者

当前前端实际源码目录为：

```text
hunyuan-design/apps/hunyuan-system
hunyuan-design/apps/web-ele
```

Codebase Memory 和源码搜索均未发现以下当前页面路径：

```text
business/oa
oa/notice
oa/enterprise
oa/invoice
oa/bank
```

但数据库有效菜单仍保存 OA 页面组件路径，主要形态为：

```text
/business/oa/notice/*.vue
/business/oa/enterprise/*.vue
```

当前前端登录适配器的行为是：

1. 将数据库菜单的 `component` 路径规范化。
2. 在本地视图路径集合中查找对应页面。
3. 命中本地页面时直接创建路由。
4. 未命中时使用模块桥接页面，而不是假定页面一定存在。

因此，当前状态存在明确的不一致：

- 后端 OA 入口仍存在。
- 数据库菜单和角色授权仍存在。
- 当前前端源码没有对应 OA 页面。
- 菜单加载机制可能将这些菜单导向桥接页，或在真实登录流程中表现为不可用入口。

这项不一致不能被解释为 OA 已经完成退役。它只能说明 OA 的前端入口责任没有完成收口，
需要单独建立入口迁移或菜单清理切片。

## 5. 运行态验证边界

本批尝试验证以下运行态入口：

- `/v3/api-docs`
- OA 代表路由和未授权请求

截至 2026-07-23，`127.0.0.1:1024` 没有监听进程，因此请求无法建立连接。
该结果只能证明当前后端未运行，不能解释为 OA 路由返回 HTTP 404，也不能作为 OA
运行态退役证据。

后续运行态验收必须在只启动端口 `1024` 对应后端 Java 进程后重新执行，范围包括：

- OpenAPI 中 OA 路径是否注册。
- 代表性 GET、OPTIONS 或未授权请求的状态码。
- 登录后菜单是否出现桥接页、空白页或可用页面。
- 通知员工查询和详情接口是否能完成只读调用。

不得为了本审计停止前端，也不得停止与端口 `1024` 无关的服务。

## 6. 处置建议

### 6.1 KEEP：通知能力

保留通知后端、通知数据和员工通知 API。下一步应补齐：

- 当前前端入口或替代入口的 owner。
- 通知列表、详情和已读/查看记录的真实浏览器流程。
- 通知可见范围与当前组织部门数据的权限验收。
- `t_notice_view_record` 为 0 的原因确认：未使用、数据未初始化，还是查看链路未接入。

在这些证据完成前，不删除通知表、通知 Controller 或通知权限。

### 6.2 ADAPT：企业、银行和发票

这些能力保留后端和数据，先不继续扩建。每个子域都需要完成以下判断：

- 是否仍属于本系统的目标业务。
- 数据 owner 是 OA、企业主数据、财务能力还是外部系统。
- 是否需要在当前前端恢复页面。
- 是否只需迁移数据和查询能力，还是仍需保留写操作。
- 是否需要把企业员工关联迁移到组织/身份能力。

如果确认没有业务 owner，应先完成数据导出、角色授权迁移和替代入口确认，
再单独提交退役切片，不与通知能力混批。

### 6.3 ADAPT：OA 菜单与权限

在页面责任未明确前，保留 OA 菜单和权限作为迁移台账，不继续新增授权。
下一批应建立菜单清理或恢复入口的二选一验收：

1. 恢复当前前端可用页面，并通过角色、直接 URL 和 API 权限验收。
2. 明确 OA 无业务 owner，完成数据备份、角色授权迁移、菜单删除和后端路由退役。

不能只删除数据库菜单而保留后端，也不能只删除后端而保留角色可见菜单。

## 7. 关闭条件

OA 整体退役前必须同时满足：

1. 通知、企业、银行、发票及企业员工关联均完成 owner 和消费者确认。
2. 开发库和历史快照中的 OA 数据完成备份、保留期限和迁移方案确认。
3. 前端菜单、组件路径、直接 URL、后端路由和 OpenAPI 形成一一对应关系。
4. 所有 OA 角色授权完成迁移或明确删除，且无有效角色继续依赖旧节点。
5. 仓库内生产消费者、测试消费者和仓库外正式消费者完成分类。
6. 代表性 API、OpenAPI、菜单加载和浏览器角色验收全部通过。
7. 退役迁移在空库和当前开发库均通过，且不改写 `V3.64.0` 历史基线。
8. Codebase Memory 刷新后，目标 Controller、Service、前端组件和旧路由达到预期归零。

在上述条件全部满足前，OA 状态保持为：

```text
通知：KEEP
企业/员工关联/银行/发票：ADAPT
菜单与组件引用：ADAPT
整体退役：NOT_READY
```

## 8. 下一步执行项

1. 仅核对并启动端口 `1024` 对应后端 Java 进程。
2. 重新执行 OpenAPI、OA 代表路由和登录后菜单加载验证。
3. 确认 OA 前端页面是否位于未被当前索引覆盖的构建产物或独立应用中。
4. 为通知建立最小只读浏览器验收。
5. 分别为企业、银行、发票建立 owner 与数据处置台账。
6. 在 owner 确认前不写 OA 退役迁移，不删除 OA 菜单、权限、表或 Controller。
