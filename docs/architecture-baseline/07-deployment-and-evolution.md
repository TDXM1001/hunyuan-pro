# 部署与演进策略

## 1. 当前部署形态

第一阶段保持简单：

```text
管理后台静态资源
        ↓
Nginx 或等价入口
        ↓
Spring Boot 应用
        ↓
MySQL 8.x
```

可以使用 Docker Compose 统一开发或部署依赖。Redis、对象存储、消息队列和搜索引擎均为按需组件，不作为当前架构成立的前提。

## 2. 仓库与发布

个人开发阶段推荐单仓库：

```text
project/
├─ frontend/
├─ backend/
├─ docs/
├─ deploy/
└─ scripts/
```

以上目录为逻辑示意，不构成对现有仓库目录的重命名要求。

前后端可以独立构建，但同一次业务修改能够在一个变更集中同步更新接口、测试和文档。

内部前端包和后端模块初期跟随产品统一版本发布，不为每个内部模块维护独立语义版本。只有包真正被多个产品独立消费时，再引入独立版本兼容策略。

## 3. 发布顺序

常规发布应具备明确顺序：

```text
备份与变更检查
  -> 构建并验证制品
  -> 执行或验证 Flyway 迁移
  -> 部署后端
  -> 部署前端
  -> 健康检查和关键流程验证
  -> 观察日志与指标
```

涉及不兼容数据库变更时，应采用展开与收缩策略，避免旧应用和新结构在滚动或回滚期间无法共存。

## 4. 可观测性

第一阶段至少需要：

- 统一结构化日志。
- 请求追踪 ID。
- 明确的启动和健康检查。
- 数据库迁移结果。
- 关键业务操作审计。
- 异常告警和基本运行指标。

日志不能记录密码、完整 Token、敏感个人信息或不必要的请求正文。

## 5. 新增终端

新增 H5、小程序或桌面端时，按以下顺序评估：

1. 新终端服务的用户和任务是否与现有管理后台相同。
2. 是否可以复用现有 API，还是需要新的查询模型和接口适配层。
3. 哪些类型、协议和无界面规则可以共享。
4. 哪些页面必须针对终端重新设计。
5. 新终端是否引入新的登录方式和安全边界。

新增终端不自动意味着新增后端服务。

## 6. 引入 Redis 的触发条件

出现以下需求时再评估 Redis：

- 分布式会话或短期凭证。
- 高频热点缓存。
- 明确的限流需求。
- 分布式锁且数据库约束不能满足。
- 临时任务状态或高频计数。

引入缓存必须同时设计失效、降级和数据一致性策略。

## 7. 引入消息队列的触发条件

出现以下证据时再评估消息队列：

- 异步任务需要削峰。
- 消费者需要独立扩缩容。
- 跨进程事件需要可靠重试。
- 多个独立系统需要订阅同一事件。
- 同步调用已经造成明显可用性问题。

单纯为了“模块解耦”不构成引入消息队列的充分理由。模块化单体可以先使用进程内事件和 Outbox。

## 8. 拆分微服务的触发条件

只有至少出现一个明确条件，并且拆分收益高于运维成本时才进行：

- 某模块需要独立扩容。
- 某模块需要独立发布节奏。
- 不同团队需要独立负责和部署。
- 故障隔离有明确业务价值。
- 安全、合规或数据隔离要求独立运行。
- 当前单体已经出现可量化瓶颈，且模块边界稳定。

拆分前应确保：

- 数据所有权已经清晰。
- 模块公开接口已经稳定。
- 跨模块事务和失败补偿可以被明确描述。
- 日志、追踪、部署和告警能力能够承担分布式复杂度。

## 9. 演进原则

```text
先解决当前真实问题
保留清晰边界
收集运行证据
在边界稳定后演进
```

未来可扩展不等于现在必须实现。预留扩展点应表现为稳定接口和隔离边界，而不是大量空模块、抽象层和未使用基础设施。
