# 技术与架构决策记录

## 1. 决策状态说明

| 状态 | 含义 |
| --- | --- |
| 已接受 | 当前架构基线的一部分，变更时需要说明原因 |
| 暂定 | 当前推荐方案，实施前结合真实需求确认 |
| 延后 | 当前信息不足，不提前决定 |
| 当前不采用 | 已讨论但当前阶段明确不引入 |

## 2. 已接受

| 决策 | 结论 | 原因 |
| --- | --- | --- |
| 首期客户端 | 只建设管理后台 | 控制首期范围 |
| 多终端方向 | 未来允许 H5、小程序和桌面端 | 通过应用和接口适配层扩展 |
| 前端组织 | 应用 + 主包 + 业务子包 | 分离装配机制和业务能力 |
| 前端集成 | 编译期集成、路由懒加载 | 满足个人开发和统一发布 |
| 后端形态 | 模块化单体 | 保留边界并降低分布式成本 |
| 模块协作 | 公开接口，不共享内部 Repository | 保护数据所有权和演进能力 |
| 当前事务 | 单数据库本地事务 | 简单、可靠、易测试 |
| 接口风格 | REST + OpenAPI | 契约清晰且工具成熟 |
| API 命名空间 | 当前使用 `/api/admin/v1` | 区分终端并保留版本边界 |
| 数据迁移 | Flyway | 数据结构变更可审计、可重复验证 |
| 数据库测试 | Testcontainers + 真实 MySQL | 避免内存数据库行为差异 |

## 4. 延后决定

| 事项 | 何时决定 |
| --- | --- |
| 真实业务模块及边界 | 完成用户、任务、流程和数据分析后 |
| MyBatis 或 JPA | 明确查询复杂度和领域规则后 |
| 多租户 | 明确产品交付模式后 |
| 认证方式 | 明确用户来源、SSO和终端登录方式后 |
| ID 策略 | 明确部署形态、数据规模和集成边界后 |
| MySQL事务隔离级别 | 有真实并发用例并完成测试后 |
| Redis | 出现明确缓存、会话、限流或协调需求后 |
| Outbox | 出现可靠事务后异步需求后 |
| 消息队列 | 出现独立消费、削峰或跨系统可靠事件后 |
| 小程序框架 | 小程序用户任务和原生能力明确后 |
| H5框架 | H5是否需要SEO、SSR和跨端复用明确后 |
| Tauri或Electron | 桌面端系统集成需求明确后 |

## 5. 当前不采用

| 方案 | 当前不采用的原因 |
| --- | --- |
| 微服务 | 没有独立扩缩容、团队自治或故障隔离证据 |
| 微前端 | 当前是个人开发和统一发布 |
| 运行时动态插件 | 没有第三方安装、独立升级需求 |
| 独立 BFF 服务 | 当前只有管理后台，接口层即可满足 |
| 多数据库 | 增加事务和运维成本，没有明确收益 |
| 默认消息队列 | 当前本地事务和事务后事件足够 |
| 所有终端共享 UI | 不同终端交互与信息密度不同 |
| ORM自动更新生产表 | 不可审计且难以安全回滚 |
| H2替代全部数据库集成测试 | 与MySQL的SQL和事务行为不一致 |

## 6. 进入详细设计的条件

架构基线完成后，下一阶段需要真实回答：

1. 系统服务哪些用户。
2. 用户需要完成哪些任务。
3. 有哪些角色、权限和数据范围。
4. 核心业务对象及生命周期是什么。
5. 哪些操作必须保持事务一致。
6. 哪些外部系统和终端需要集成。
7. 性能、安全、审计、备份和可用性目标是什么。

只有得到这些信息，才能确定业务模块、数据模型、接口和页面。架构示例不能替代需求分析。

## 7. 变更规则

架构决策允许演进。发生变更时至少记录：

- 当前问题和证据。
- 原决策为什么不再适用。
- 新方案的收益、成本和风险。
- 数据、接口、部署和回滚影响。
- 验证新决策是否有效的方法。
