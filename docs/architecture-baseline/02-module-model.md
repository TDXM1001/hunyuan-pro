# 模块模型与边界

## 1. 统一术语

| 术语 | 定义 |
| --- | --- |
| 应用（Application） | 可以独立运行、构建和部署的产品，如管理后台或未来的小程序 |
| 主包（App Shell） | 应用的组合入口，负责启动、路由、身份、权限、布局和模块装配 |
| 平台包（Platform Package） | 提供 HTTP、UI、认证、日志等非具体业务能力的前端包 |
| 业务子包（Feature Package） | 面向一组完整用户任务的前端业务能力包 |
| 后端模块（Backend Module） | 拥有业务规则、用例、数据和公开接口的后端边界 |
| 能力（Capability） | 可以授权、校验和审计的稳定操作编码 |
| 接口适配层（Interface Adapter） | 将不同终端协议转换为应用用例的入口层 |

前端业务子包和后端模块不要求一一对应。前端按照用户任务组织界面，后端按照业务规则和数据所有权组织能力。

## 2. 模块边界如何产生

模块不能在没有需求的情况下凭技术想象创建。识别候选模块时，应依次分析：

1. 系统服务哪些用户和角色。
2. 用户需要完成哪些完整任务。
3. 哪些业务概念拥有独立生命周期和规则。
4. 哪些数据由同一组规则创建和修改。
5. 哪些变化需要在同一事务中保持一致。
6. 哪些能力可以通过稳定接口对外提供。

如果两个候选模块长期相互修改对方数据、共享大量内部对象或形成循环依赖，通常说明边界划分错误。

## 3. 子包不是部署单元

当前采用以下模型：

```text
开发时：按子包隔离
构建时：所有子包一起编译
部署时：形成一个管理后台应用
运行时：按模块、配置和权限决定可用范围
```

需要区分四种边界：

| 边界 | 当前策略 |
| --- | --- |
| 代码边界 | 子包和后端模块独立 |
| 构建边界 | 统一构建 |
| 部署边界 | 前端一个管理后台产物，后端一个应用 |
| 启用边界 | 可以按模块和能力控制 |

一个模块可以已经包含在构建产物中，但对当前部署、租户或用户不可用。这不等于运行时安装或卸载代码。

## 4. 模块与能力标识

模块和能力应使用稳定、与页面路径无关的编码：

```text
模块：module.a

能力：
module.a.read
module.a.create
module.a.update
module.a.operation-x
```

真实业务确定后，使用真实领域名称替换 `module.a`。编码一旦对外用于权限、审计或接口，不应因页面改名随意变化。

模块启用与用户权限必须分开：

```text
模块启用：产品、部署或租户是否拥有该能力
用户权限：当前用户是否可以执行模块内的具体操作
数据权限：用户可以操作哪些具体数据
```

## 5. 模块依赖

模块可以声明依赖，但应遵守：

- 依赖方向清晰且可形成有向无环图。
- 模块只能调用依赖模块公开接口。
- 不允许引用其他模块内部实体、Mapper、Repository 或数据库表完成写操作。
- 依赖缺失时应在启动或模块装配阶段报告明确错误。
- 出现循环依赖时，优先重新检查业务边界，而不是增加全局事件总线绕过问题。

```text
允许：模块 B -> 模块 A 的公开 Facade
禁止：模块 B -> 模块 A 的 Repository 或内部 Entity
```

## 6. 公共能力边界

`shared` 或 `shared-kernel` 必须保持很小，只承载含义稳定且已有多个明确使用方的内容，例如：

- ID、时间、分页等基础类型。
- 当前用户抽象。
- 统一错误结构。
- 领域事件基础接口。

不应放入：

- 所有模块共同继承的万能业务实体。
- 万能 Service 或 Repository。
- 尚未证明可复用的业务 DTO。
- 仅因为“不知道放在哪里”而抽取的工具类。

## 7. 模块启停

运行时关闭模块意味着：

- 不提供该模块的页面、菜单和操作入口。
- 后端拒绝调用该模块能力。
- 相关定时任务和事件处理根据设计停止。
- 保留该模块已有数据。

关闭模块不应自动删除表或业务数据。数据库迁移跟随产品版本统一执行，不在用户点击启用模块时临时建表。

## 8. 何时才需要动态插件或微前端

只有出现以下需求时，才评估运行时插件：

- 客户能够自行安装和卸载功能模块。
- 模块需要独立收费、上传和升级。
- 第三方开发者需要提供模块。
- 主应用不重新发布也必须替换模块代码。

只有多个团队必须独立构建和发布前端时，才评估微前端。当前个人开发和统一发布场景不采用这两种方案。
