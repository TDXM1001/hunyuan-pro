# 数据、接口与安全基线

## 1. MySQL 基线

当前数据库候选确定为 MySQL 8.x：

- 新部署环境可评估 MySQL 8.4 LTS。
- 现有云服务或基础设施以 MySQL 8.0 为主时，可以使用稳定的 8.0 版本。
- 存储引擎统一使用 InnoDB。
- 字符集统一使用 `utf8mb4`。
- 表名和字段名统一使用小写 `snake_case`，避免 Windows 与 Linux 大小写行为差异。

数据库排序规则需要根据唯一性和检索语义选择，不能默认认为中文、英文大小写和业务编码具有同样比较规则。

## 2. 数据所有权

每张业务表有唯一所属模块：

```text
所属模块：可以读写
其他模块：通过公开接口操作
跨模块报表：允许只读聚合
```

不建立所有模块共同修改的“万能业务表”。跨模块引用可以保存目标 ID 和必要业务快照，但不能借外键关系绕过目标模块规则修改其数据。

ID 策略应在部署方式和数据规模明确后统一选择。未确定前，不同时混用自增 ID、UUID 和雪花 ID。

## 3. Flyway 迁移

生产数据库结构由 Flyway 管理：

```text
db/migration/
├─ V1__baseline.sql
├─ V2__create_module_a_tables.sql
└─ V3__add_module_a_indexes.sql
```

规则：

- 已发布迁移不可随意修改，应新增后续迁移修正。
- 应用启动前或发布流程中执行迁移并验证结果。
- 生产环境禁止 ORM 使用 `ddl-auto=update` 修改结构。
- 使用 JPA 时，生产环境可以使用 `ddl-auto=validate` 检查映射。
- 模块关闭不回滚迁移，也不删除表和数据。

Spring Boot 项目通常需要 MySQL JDBC 驱动、`flyway-core` 和与版本对应的 MySQL Flyway 支持模块。

## 4. 时间、事务与并发

时间字段应表达清楚业务语义，例如：

```text
created_at
updated_at
completed_at
```

可以使用支持毫秒精度的 `DATETIME(3)`。如果系统未来跨时区，持久化、JVM、数据库连接和序列化应统一使用 UTC，展示层再转换用户时区。

MySQL InnoDB 的事务隔离级别必须明确记录。默认 `REPEATABLE READ` 与可选的 `READ COMMITTED` 在锁和一致性方面不同，应通过实际并发用例和集成测试确定，而不是仅复制模板。

核心状态转换应考虑条件更新、唯一约束、版本号或幂等键，防止重复提交和并发覆盖。

## 5. API 分层与命名

当前管理后台接口：

```text
/api/admin/v1/**
```

未来按真实终端任务增加：

```text
/api/app/v1/**
/api/open/v1/**
```

H5 和小程序如果面向相同用户与任务，可以共用 `App API`，不要求每个客户端都拥有一套独立接口。

接口应表达业务动作，而不是允许前端任意修改状态字段：

```text
推荐：POST /resources/{id}/operation-x
谨慎：PUT /resources/{id} 并由前端直接指定内部状态
```

查询接口可以针对终端提供合适的组合结果。管理后台详情和移动端摘要不需要强行使用同一个万能 DTO。

## 6. 统一接口契约

API 基线应统一：

- 请求和响应字段命名。
- 分页、排序和筛选格式。
- 错误码、错误消息和追踪 ID。
- 日期、时间和时区格式。
- 枚举及未知值处理。
- 文件上传和下载协议。
- 幂等请求规则。
- OpenAPI 文档生成和兼容性检查。

后端生成 OpenAPI，前端由契约生成 TypeScript 类型和客户端代码。生成代码单独存放，业务子包不得直接修改生成文件。

## 7. 身份模型

不同终端可以采用不同登录方式，但进入业务层前应转换为统一主体：

```text
Principal
├─ userId
├─ organizationId（如需要）
├─ tenantId（如需要）
├─ clientType
├─ roles
├─ capabilities
└─ dataScope
```

`clientType` 可以预留管理后台、H5、小程序、桌面端和开放接口等类型，但当前只实现真实需要的管理后台认证流程。

业务模块不直接理解微信 `openid`、浏览器 Cookie 或桌面设备标识，这些属于认证适配层。

## 8. 授权层次

授权至少区分：

```text
模块启用：系统或租户是否拥有该模块
页面能力：用户是否可以访问某类资源
操作能力：用户是否可以执行具体动作
数据权限：用户可以操作哪些数据
业务规则：目标当前状态是否允许操作
```

前端根据能力过滤入口，后端对每次请求重新验证。直接访问 URL 或调用接口不能绕过授权。

多租户、SSO、OAuth/OIDC、Token 形式和刷新机制均需结合实际产品形态单独设计，当前架构不预先假设。
