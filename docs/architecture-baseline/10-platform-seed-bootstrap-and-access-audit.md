# 平台 Seed、管理员 Bootstrap 与权限盘点

## 1. 阶段结论

`V3.64.0` 继续保持纯结构基线，`V3.65.0` 开始承载经过复核的平台种子数据。两者边界如下：

```text
V3.64.0 schema baseline
    -> V3.65.0 non-sensitive platform seed
    -> environment-driven initial administrator bootstrap
    -> operator login and mandatory password change
```

Flyway 不保存用户凭据。初始管理员只在显式启用 bootstrap 且提供环境变量密码时创建；密码经现有 Argon2 机制加密，应用日志和 bootstrap 审计表均不记录明文或密文。

本阶段不复制开发库，也不把示例业务数据伪装成平台基线。

## 4. V3.65.0 Seed 范围

### 4.1 配置

- `super_password`：空值，默认关闭万能密码路径；不会覆盖已有环境值。
- `level3_protect_config`：登录失败 5 次锁定 30 分钟、30 分钟无操作冻结、密码复杂度开启、3 个月换密、最近 5 次不可重复、文件检测开启、上传上限 30 MB、双因子默认关闭。

### 4.2 字典

- `SYS_GENDER`：与后端 `GenderEnum` 的 0/1/2 对齐。
- `SYS_DISABLED_FLAG`：0 启用、1 禁用。

`GOODS_PLACE` 明确排除，它只服务商品示例。

### 4.3 菜单

平台 seed 只有三组菜单：

| 目录 | 页面 |
| --- | --- |
| 组织架构 | 部门、职务、员工、角色 |
| 系统设置 | 菜单、参数、字典 |
| 安全治理 | 安全基线、登录失败锁定、登录日志、操作日志 |

所有页面组件都在当前前端仓库真实存在。功能点权限与后端注解使用同一编码。迁移通过语义键收敛空库和已有 `V3.64.0` 数据库，不依赖开发库菜单主键。

### 4.4 角色与数据范围

- 内置角色：`platform_admin` / 平台管理员。
- 菜单权限：授予本次 seed 的全部目录、页面和功能点。
- 数据范围：当前系统只定义“系统通知”范围，平台管理员为全部数据（`data_scope_type=1`，`view_type=10`）。
- 初始管理员：同时设置 `administrator_flag=1` 并关联 `platform_admin`，防止角色表暂时异常时失去恢复入口。

## 5. 初始管理员 Bootstrap

Bootstrap 默认关闭。首次启动新环境前设置：

```powershell
$env:HUNYUAN_FLYWAY_ENABLED = 'true'
$env:HUNYUAN_BOOTSTRAP_ADMIN_ENABLED = 'true'
$env:HUNYUAN_BOOTSTRAP_ADMIN_LOGIN_NAME = 'admin'
$env:HUNYUAN_BOOTSTRAP_ADMIN_PASSWORD = '<由部署人员生成的强密码>'
$env:HUNYUAN_BOOTSTRAP_ADMIN_ACTUAL_NAME = '系统管理员'
```

只有已有库存在多个根部门且尚无管理员时，才需要额外指定：

```powershell
$env:HUNYUAN_BOOTSTRAP_ADMIN_DEPARTMENT_ID = '<有效部门ID>'
```

执行规则：

1. 未显式启用时不访问或修改管理员数据。
2. 启用但缺少强密码时启动失败，不生成默认密码。
3. 同名有效管理员已存在时只核对平台角色并记录幂等审计。
4. 已有其他有效管理员时拒绝创建第二个 bootstrap 管理员。
5. 创建成功后写入 `t_bootstrap_audit`，审计内容不含任何凭据。
6. 初始密码被标记为已过期，首次登录必须修改。
7. 首次成功后应移除密码环境变量并关闭 `HUNYUAN_BOOTSTRAP_ADMIN_ENABLED`。

## 6. 数据约束

`V3.65.0` 增加以下唯一约束，防止 bootstrap 和授权关系并发重复：

- `t_config(config_key)`
- `t_employee(login_name)`
- `t_role_menu(role_id, menu_id)`
- `t_role_data_scope(role_id, data_scope_type)`
- `t_dict_data(dict_id, data_value)`

已有数据库启用 Flyway 前必须先执行重复数据预检；迁移遇到重复时应失败并由人工确定保留记录，不能在迁移脚本里静默删数据。
