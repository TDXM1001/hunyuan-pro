# Hunyuan Pro 项目级 Agent 开发规则

## 1. 权限与硬约束

- 本文件只保存长期稳定的项目级 Agent 指令，包括仓库约定、稳定命令、验证步骤和审查期望；领域目标、动态入口、端口、本机路径和一次性交付清单放入对应领域文档或运行手册。
- 默认目标是用最短的安全路径，把已确认需求变成可运行、可验证、可解释的结果；流程步骤齐全不等于交付完成。
- 安全、数据完整性、审计和已确认的可恢复性是交付硬约束，不得因进度或便利而豁免；改变这些约束必须进入设计交付并由用户明确决策。
- 在不违反硬约束的前提下，规则冲突依次遵循：用户最新要求、本文件、已确认设计或决策、相关领域文档、仓库代码与测试模式、Agent 内部清单与 Skill 默认流程。
- 解释、审查、诊断和状态报告保持只读；只有实现、修复或用户明确要求修改时才编辑文件。
- Skill 和多 Agent 只用于提高质量或速度，不得扩大范围、增加批准关卡、引入依赖或强制生成过程文档。

## 2. 任务分类与交付路径

- 开发任务只分类一次。若新证据没有改变公共契约、数据语义、安全权限、模块边界、不可逆迁移或依赖选择，不得在实施中切回设计流程。
- 开始编辑前，在 Agent 内部明确本次可观察行为、影响边界和最低验证证据；小任务不为此创建文档，已有设计的验收标准直接复用。
- **直接交付**：需求明确、影响局部、可逆且不涉及上述重大变化。检查工作树和最少相关上下文后直接实现，只做必要的聚焦验证；不创建设计、计划或审查工件。
- **设计交付**：用户明确要求设计，或存在无法从仓库消解且会实质影响上述重大事项的不确定性。使用真正相关的最少 Skill 形成一份整体设计，最多进行一次合并审查；用户已经确认时审查预算视为已消费。
- 用户指定的当前设计或基线能够回答目标、边界、关键语义、主要风险和验收方式时，直接以其为任务契约；只核对状态、新鲜度、直接上游和硬约束冲突，不重新设计。
- 任务只要求设计时，交付设计后停止。任务同时包含实施时，设计确认后连续实施，不再按文件、接口、页面、测试或内部批次逐点审批。
- 只有重大事项需要用户决策、用户改动形成无法绕开的冲突，或必要权限与环境无法在当前范围解决时才暂停；其余普通实现细节和可逆选择由 Agent 按仓库模式自主完成。

## 3. 实施与反馈节奏

- 默认交付单位是可独立验收的纵向闭环，可跨数据库、后端、API、前端、测试和文档；不按技术层机械拆分，也不把字段、文件或断言变成独立关卡。
- 跨层交付先打通最短可运行主链并取得第一份跨层证据，再在同一确认范围内补齐异常、兼容和治理能力。
- 实施拆分是 Agent 内部清单。设计已包含交付批次和验收标准时，不再编写详细实施计划；只有用户要求，或不存在合适事实源且确有跨会话交接价值时，才持久化唯一执行清单。
- 同一交付块只保留一个持久化事实源，不同时堆叠设计、详细计划、子任务 brief、执行 report 和 review package；使用持久化清单时原位更新进度和证据。
- 实施期间只在自然批次边界运行能够阻止错误继续扩散的低成本检查，例如编译、类型检查、关键算法或状态机测试、API 契约检查和当前主链聚焦测试；这些检查不是新的审批关卡。
- 缺陷、关键算法、状态机、并发、权限和数据契约应先建立失败证据；普通映射、胶水代码和成熟模式下的常规实现可以先实现再聚焦验证。
- 若阅读、规划或过程文档持续增长却没有新增实现、失败证据或运行证据，立即停止扩展上下文并转向下一项可验证改动；无法转向时只报告真实阻塞。

## 4. 仓库与实现边界

- 默认在用户当前分支工作；保留用户和其他任务的未提交修改，不回退、不覆盖、不整理当前交付块之外的内容。
- 代码、契约、迁移、权限和测试放入既有所有权模块，优先复用仓库模式和公共组件；未经用户明确批准不得新增依赖。
- 参考仓库只用于理解机制、交互和契约语义，不得整包迁移代码、命名、依赖假设或模块边界。
- 相邻问题只记录，不顺带扩展重构。并行任务必须有独立所有权边界，同一文件或强耦合链路保持单一负责人。
- 临时协调 brief、执行 report、review package、运行日志、截图和测试输出不得提交仓库，除非用户明确要求作为长期证据；暂存和提交只包含当前交付块的文件。

## 5. 工程一致性约束

- 实现前先识别受影响模块的现有规范、相邻实现和公共组件；已有模式能够满足需求时必须复用，不为单个页面或接口建立平行实现。
- 前端依赖版本以 `hunyuan-design/package.json`、`hunyuan-design/pnpm-workspace.yaml` 和锁文件为唯一事实源，不在本文件复制版本号，未经用户明确批准不得升级或新增依赖。
- 普通列表页面遵循 `docs/frontend-list-table-page-standard.md`；编辑与详情页面遵循 `docs/frontend-edit-detail-page-standard.md`；弹窗与抽屉表单遵循 `docs/frontend-dialog-drawer-form-standard.md`。
- 前端优先使用仓库公共组件：列表使用既有搜索、表格和分页组件；页面级编辑优先使用 `ArtEditPage`、`ArtEditSection`；页面级详情优先使用 `ArtDetailPage`、`ArtDetail`。
- 短小、聚焦且完成后立即返回来源列表的操作使用弹窗或抽屉；字段多、分区多、包含子表或附件、状态复杂、需要独立路由或大量上下文的业务使用单独编辑页和详情页。详情保持只读，编辑通过明确入口进入。
- 后端版本和技术栈以 `hunyuan-backend/pom.xml` 及所属模块配置为事实源；新增代码遵循所属模块现有的 Controller、Service、DAO、Entity、Form、VO、统一响应、异常、校验、事务、权限和审计模式，不跨层绕过既有职责。
- 数据库结构或基础配置变化必须在 `数据库SQL脚本/mysql/sql-update-log/` 中新增版本化增量 SQL，不以修改全量初始化脚本作为已有环境的升级方式。
- 增量 SQL 必须覆盖功能实际依赖的完整配置，包括适用的表与字段、初始化数据、菜单与动态路由、按钮或接口权限、字典与字典项；不涉及的类别不得机械新增。迁移必须说明执行前提、兼容影响和必要的恢复方式。
- 菜单或动态路由只有在对应前端路由和页面可用时才能加入；前端按钮权限、后端接口权限和角色授权配置必须使用一致的权限语义，不得产生死菜单、空路由或只有单侧生效的权限。

## 6. 最终验证与完成声明

- 按风险所需的完整回归、跨层集成、浏览器业务流和真实运行验收集中在交付块末尾统一执行；发现问题后集中修复，只重跑受影响检查和最终门禁，最终状态未变化时不重复运行。
- 验证强度与风险成比例：纯文档检查差异、结构、链接和 UTF-8；局部行为检查原问题、聚焦测试和相关门禁；跨层交付证明主路径和受影响异常；平台、公共语义、数据、安全或迁移还必须覆盖影响、权限或数据边界、关键失败路径、兼容或恢复、相关回归和真实运行。
- 审查解决重大决策，验证检查实现正确性，验收证明业务闭环；测试、代码检查和完成前验证不得重开设计批准。
- 平台与语义变更的执行证据采用追加记录或版本化链接，保留日期、仓库版本、基线版本、实际证据和未关闭缺口，不得覆盖历史执行事实。
- 必要门禁被无关基线或外部环境阻塞时，完成最大可行的聚焦验证并在事实源记录缺口、影响、替代证据和关闭条件，状态标记为 `NOT_RELEASABLE`，不得进入可发布基线或宣称模块关闭。
- 不以推测或历史通过结果代替本次证据。完成时只报告实际变更、当次验证、剩余风险和文档状态；“实现完成”和“模块关闭”必须明确区分。
